ALF-123 — Back up database daily

Refinement Ops · CI GitHub Actions · R2 Spec only — no implementation. Build in a later session.

1 · Context & problem

alfred runs on the Supabase free tier, which has no automated backups. The database (items, folders, projects, epics, code_items, …) is the single-user's whole task & code-factory state — losing it is unrecoverable. Our schema lives in git (database/migrations/), so it can always be rebuilt; the data cannot.

The spike docs/scheduled-cloud-backups.md already investigated this and landed the architecture: a scheduled GitHub Actions workflow takes a daily logical dump with supabase db dump (a hardened pg_dump wrapper), gzips it, and uploads it to a Cloudflare R2 bucket with lifecycle-based retention — effectively $0, using only providers already in our stack. This spec turns that decision into a buildable change. Read the spike first; this spec does not re-argue provider choices already settled there (R2 vs Glacier, Actions vs a Cron Worker, R2 vs AWS/CDK).

2 · Decisions (locked during refinement)

The spike left five open questions “for the spec.” They are resolved here:

Dump type
Full logical dump (schema + data), not data-only. Each artifact is then self-contained and restorable without replaying migrations. Migrations remain the authoritative schema; the dump is a self-sufficient recovery artifact, not a schema source.
Retention
Two tiers, both by R2 object-lifecycle rule (not scripted deletion): 30 rolling dailies under a daily/ prefix (expire after ~35 days), plus monthly snapshots kept indefinitely under a monthly/ prefix (no expiry). Each run writes its dump to both a dated daily key and a year-month monthly key; the monthly key overwrites within the month, so it settles to that month's last good backup and freezes once the month rolls over (§4.2). No weekly rollups in v1 (deferred, §6).
Encryption
R2 server-side encryption at rest (default, automatic) + TLS in transit. No client-side gpg in v1 — deferred as defense-in-depth (§6). Credentials never leave GitHub Actions secrets.
Verification
Every nightly self-verifies: the same job restores its fresh dump into a throwaway Postgres and asserts core tables load with rows, before the run is considered green. A backup unverified against a restore is a hope, not a backup (spike). Reuses the throwaway-cluster technique the database/ slow suite already relies on.
Failure alerting
GitHub's built-in failed-scheduled-run email to the repo owner is the v1 alert. Telegram wiring (from the notifications spike) stays a separate, later, optional feature — the seam is noted but not built here (§6).
Not covered by the back-pressure gates. A workflow under .github/ is not linted or type-checked by check:fast/check:slow (spike “Operational notes”). So the rule for this ticket: keep the YAML thin and push every scrap of real logic into a committed, testable script under database/ (§4.2) — a unit test can cover that script; untested inline bash accretes into a silently-broken nightly.

3 · How it works

4 · Proposed change

Three committed pieces — the workflow, a testable dump/verify script, and its unit test.

4.1 · The scheduled workflow

New file .github/workflows/backup.yml, sibling to ci.yml. It mirrors ci.yml's shape (actions/checkout@v4 → actions/setup-node@v4 with node-version-file: '.nvmrc' → npm ci) and then runs the committed script. The YAML stays thin: scheduling, a Postgres service for the verify step, secret wiring, and one run: that calls the script.

.github/workflows/backup.yml (new) — shape, fleshed out at build time 
name: Backup

on:
  schedule:
    - cron: '17 9 * * *'   # daily; off-the-hour to dodge scheduler contention (spike)
  workflow_dispatch: {}     # manual runs / restore drills

# Only one backup at a time; a slow run must not overlap the next day's.
concurrency:
  group: db-backup
  cancel-in-progress: false

jobs:
  backup:
    runs-on: ubuntu-latest
    services:
      verify-db:                     # throwaway Postgres for the in-job restore check
        image: postgres:16
        env: { POSTGRES_PASSWORD: postgres }
        ports: ['5432:5432']
        options: >-
          --health-cmd "pg_isready" --health-interval 5s
          --health-timeout 5s --health-retries 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version-file: '.nvmrc', cache: 'npm' }
      - run: npm ci
      - uses: supabase/setup-cli@v1
      - name: Dump, verify-restore, upload
        run: node database/src/backup.ts
        env:
          SUPABASE_DB_URL:       ${{ secrets.SUPABASE_DB_URL }}
          VERIFY_DB_URL:         postgres://postgres:postgres@localhost:5432/postgres
          AWS_ACCESS_KEY_ID:     ${{ secrets.R2_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.R2_SECRET_ACCESS_KEY }}
          R2_BUCKET:             ${{ secrets.R2_BUCKET }}
          R2_ENDPOINT:           ${{ secrets.R2_ENDPOINT }}
Connection string — the non-obvious one (spike gotcha). SUPABASE_DB_URL MUST be the Supabase session-mode pooler (IPv4, port 5432). Not the direct connection (IPv6-only on free tier → Actions runners are IPv4-only, it just won't connect) and not the transaction pooler (port 6543 → doesn't support pg_dump). This belongs in the secret's value and the setup runbook (§4.4).

4.2 · The dump / verify / upload script

All real logic lives here so it's testable — the spike's explicit instruction. A small Node script under database/src/ (TypeScript, matching run.ts/apply.ts), run with node database/src/backup.ts. It orchestrates three shell steps and fails hard if any step or assertion fails.

database/src/backup.ts (new) — responsibilities 
// 1. DUMP:   supabase db dump --db-url $SUPABASE_DB_URL -f - | gzip > backup.sql.gz
//            (full logical dump; assert the gzip is non-empty / above a floor size).
// 2. VERIFY: gunzip -c backup.sql.gz | psql $VERIFY_DB_URL   (restore into the throwaway),
//            then query the restored DB and assert CORE TABLES EXIST and the dump is
//            structurally sane — e.g. `items`, `folders`, `projects` are present. This catches a
//            silently-truncated / unrestorable dump the same night.
// 3. NAME +  Upload the SAME verified gzip to TWO keys (UTC dates, read at runtime — not committed):
//    UPLOAD:    • daily/$(date -u +%Y-%m-%d).sql.gz    — one slot per day; a same-day re-run
//                 overwrites, so re-runs never accumulate dupes.
//               • monthly/$(date -u +%Y-%m).sql.gz     — one slot per month; every daily run
//                 overwrites it, so it settles to the month's LAST good backup, then freezes when
//                 the month rolls over. Retention diverges purely by prefix lifecycle rule (§4.4).
//            aws s3 cp backup.sql.gz "s3://$R2_BUCKET/<key>" --endpoint-url "$R2_ENDPOINT"  (×2)
//
// Ordering is load-bearing: VERIFY must pass BEFORE either UPLOAD. Any failure → non-zero exit →
// red run → GitHub emails the owner. Dates are read at runtime (Actions clock), not committed.

Why a script, not inline bash: §4.3 can unit-test the pure parts (object-key formatting, the min-size floor, the “which tables must exist” list, argument assembly) without hitting the network — turning the one part of this feature that can regress silently into something the fast suite covers. Shelling out to supabase/psql/aws is fine; the decisions around them are what get tested.

4.4 · One-time setup (runbook, committed as docs)

None of this is code, but the feature is inert without it, so the implementation PR records the steps (extend database/README.md or a short docs/ops/ note):

  1. Create the R2 bucket; add two lifecycle rules: expire objects under the daily/ prefix after ~35 days; add no expiry rule for the monthly/ prefix (kept indefinitely).
  2. Create an R2 API token (S3 credentials) scoped to that bucket.
  3. Add GitHub Actions secrets: SUPABASE_DB_URL (session pooler, §4.1 callout), R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY, R2_BUCKET, R2_ENDPOINT.
  4. Trigger workflow_dispatch once to prove the path end-to-end (this is the demo run).

5 · Restore procedure (documented, for a real recovery)

A backup nobody knows how to restore is worthless. The runbook (§4.4 doc) states the recovery path explicitly: download the chosen object from R2 — a recent day from daily/, or an older month from monthly/ — then gunzip -c <date>.sql.gz | psql "<target-db-url>". Because the dump is full (§2), this reconstructs schema + data with no migration replay. The nightly verify step (§4.2) exercises exactly this restore path every day, so the documented procedure is continuously proven, not aspirational.

6 · Out of scope / open questions

  • Weekly rollups — v1 keeps 30 rolling dailies plus indefinite monthlies (§2). An intermediate weekly tier (e.g. keep 1/week for a year) is a later lifecycle-rule addition if the daily→monthly granularity gap ever matters.
  • Client-side encryption (gpg before upload) — deferred defense-in-depth. R2 server-side encryption covers at-rest in v1.
  • Telegram failure alerting — the notifications-spike Telegram channel is the obvious richer sink, but stays independent and optional. v1 relies on GitHub's failed-run email.
  • Supabase Storage buckets — none are in use today, so file-storage backup is explicitly out of scope until a bucket exists.
  • Point-in-time recovery / more-than-daily cadence — daily logical dumps only; PITR is a paid-tier Supabase feature, out of scope at the free-tier constraint that motivates this.
  • Scheduled-cron caveats (accepted, not solved) — GitHub may delay a scheduled run under load (fine for a daily), and 60 days of repo inactivity auto-disables scheduled workflows (a non-issue for an active repo). No mitigation built.

7 · Acceptance criteria

  • .github/workflows/backup.yml exists: scheduled daily (cron: '17 9 * * *') plus workflow_dispatch, with a concurrency group so runs don't overlap.
  • All dump/verify/upload logic lives in a committed database/src/backup.ts (thin YAML, testable script) — not inline bash in the workflow.
  • The job produces a full logical gzipped dump via supabase db dump using the session-mode pooler connection string.
  • Before upload, the dump is restored into a throwaway Postgres and asserted to contain the core tables (items, folders, projects); a dump that fails to restore fails the run and is not uploaded.
  • On success the verified dump lands in R2 under two keys: daily/YYYY-MM-DD.sql.gz and monthly/YYYY-MM.sql.gz (UTC); a same-day re-run overwrites rather than duplicates, and the monthly key overwrites within its month.
  • Retention is two R2 lifecycle rules — daily/ expires after ~35 days, monthly/ has no expiry (kept indefinitely) — documented in the runbook, with no deletion scripting in the workflow.
  • Credentials are read only from GitHub Actions secrets; none are committed or echoed to logs.
  • database/src/backup.test.ts unit-tests the pure helpers (daily and monthly object-key formatting, empty-dump guard, core-table list) and passes in check:fast.
  • A one-time setup + restore runbook is committed (in database/README.md or docs/ops/), including the exact restore command.
  • A workflow_dispatch run against real secrets is green and the dated object is visible in R2 (captured in the demo doc).
