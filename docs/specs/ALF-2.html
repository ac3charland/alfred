1 Context & problem

Alfred has grown two modules — Tasks (the inbox / folders / completed views) and Code (the Software Factory board of stories) — switched between with the header's Tasks ⇄ Code segmented control. There is today no way to jump straight to a known item. To reach a task you scan its folder; to reach a story you switch to Code, pick the project, expand the epic, and find the card. At a few hundred items that friction adds up and works against the system's "capture first, low cognitive load" principle.

ALF-2 adds a single global search in the top bar, summoned with ⌘P (macOS) / Ctrl P (Windows/Linux), that searches across both tasks and stories at once and navigates to whatever you pick. It is the keyboard-first "go to anything" affordance the two-module shell now needs.

Grounding. Both modules' data is already eagerly fetched and seeded into client stores once, at app/(shell)/layout.tsx (getAllItems() + getCodeStories()). Search therefore needs no new API or query — it reads the stores that are already in memory and filters client-side, exactly the house "fetch-all then filter client-side" pattern (see the data-flow skill).

2 What "tasks" and "stories" are here

The ticket says "searches through both tasks and stories." In this codebase those are two distinct shapes:

ConceptWhat it isStore hookSearchable fields
TASK An items row that is not in the factory (item_type = task / unclassified / a code item not yet sent). Includes subtasks and completed items. Type Item in lib/types.ts. useTasks() → Item[] title, notes
STORY A code story — an items row with item_type='code' plus a code_items sidecar, surfaced through the v_code_stories view. Type CodeStory. Carries a human ref like ALF-42. useCodeStories() → CodeStory[] (currently module-private) title, notes, ref

Both collections live under the same (shell) provider tree, so a palette mounted in AppShell can read both with no extra plumbing. useTasks() already returns the full set (active and completed — the per-view filters split them), so a single read covers everything searchable.

One small store change. useCodeStories() exists in lib/stores/code-store.tsx but is not exported (the board consumes it via useProjectBoard). This spec exports it (or a thin useSearchableStories() wrapper) so the palette can read the flat story list.

3 Proposed change

Five focused additions, all client-side, all within the existing shell:

  1. Global shortcut. A window keydown listener for (meta||ctrl)+p that preventDefault()s (overriding the browser's Print binding) and toggles the palette open.
  2. Search store. A tiny SearchProvider (built with the repo's create-context-pair helper) holding just { open, query } with open() / close() / setQuery() actions.
  3. Search palette. A Radix Dialog (components/shell/search-palette.tsx) rendered once in AppShell. It reads useTasks() + useCodeStories(), filters by query, and renders grouped, keyboard-navigable results.
  4. Trigger affordance in the top bar. A small "Search… ⌘P" button in the header (and inside the mobile hamburger) so the feature is discoverable, not just a hidden shortcut.
  5. Result navigation. Selecting a result routes to it: a task → its containing view (inbox / folder / completed) with the row scrolled-to and briefly highlighted; a story → /code/[projectId]?story=<ref>, with the board reading the story param to open its detail modal.

No backend, schema, or API change. The palette reuses existing atoms (Dialog, Badge, Input), the existing client-nav primitive (history.pushState via the ViewLink convention), and the existing custom-event pattern (ALFRED_CAPTURE_FOCUS_EVENT) for cross-component focus.

Why ⌘P specifically. The ticket asks for it. ⌘P is the browser's native Print shortcut, so the handler must call preventDefault() to claim it. This is a deliberate, documented override; ⌘K is the more conventional palette key and is raised as an open question in §13 — but the default per the ticket is ⌘P.

4 Data flow

The palette is a pure read over data already in memory; the only writes are URL navigation and a transient focus event.

7 Matching & result model

Matching is simple, instant, client-side substring matching — no server FTS, no fuzzy library (those are §13 out-of-scope). A normalized (trimmed, lowercased) query is tested against each candidate's fields.

  • Tasks match on title and notes.
  • Stories match on title, notes, and ref (so typing ALF-31 finds it directly).
  • Ranking within each group: title prefix match, then title substring, then notes-only — then by recency (created_at desc, the store's existing order). Ref-exact stories sort to the very top of Stories.
  • Caps: show up to 8 per group; if more match, show a muted "+N more — keep typing" line. Capping is surfaced, never silent.
  • Empty query: no results list; show a one-line hint ("Search tasks and stories by title, notes, or ref"). (Optionally seed with a few most-recent items — deferred, see §13.)

A unified result type keeps rendering and keyboard nav uniform across the two sources:

// components/shell/search-results.ts
export type SearchResult =
  | { kind: 'task'; id: string; title: string; subtitle: string; href: string; item: Item }
  | { kind: 'story'; id: string; title: string; ref: string; subtitle: string; href: string; story: CodeStory };

// Pure, unit-testable: store snapshots in → ranked, capped results out.
export function buildResults(query: string, tasks: Item[], stories: CodeStory[]): {
  tasks: SearchResult[]; stories: SearchResult[]; truncated: { tasks: number; stories: number };
} { /* normalize, filter, rank, cap */ }
Keeping buildResults a pure function (no React, no DOM) is the testing seam: the ranking/cap/ref-match rules get exhaustive unit tests, and the component just renders its output.

8 Selecting a result (navigation)

Tasks → containing view + highlight

A task's destination view is derived from its own fields, reusing the existing routes:

ConditionDestination
status === 'completed'/completed
folder_id set/folders/[folder_id]
otherwise (Inbox)/?view=inbox

A subtask (parent_id set) resolves to its top-level ancestor's view. After history.pushState to that view, the palette closes and dispatches a focus event so the target list scrolls the row into view and applies a brief highlight — reusing the existing custom-event convention (ALFRED_CAPTURE_FOCUS_EVENT in components/tasks/alfred-link.tsx): 

// new sibling of ALFRED_CAPTURE_FOCUS_EVENT
export const ALFRED_FOCUS_ITEM_EVENT = 'alfred-focus-item';

// on task selection:
globalThis.history.pushState(null, '', href);        // client-side view switch (ViewLink convention)
close();                                          // clear + close palette
globalThis.dispatchEvent(new CustomEvent(ALFRED_FOCUS_ITEM_EVENT, { detail: { id } }));

The task list (or row) listens for the event, and when the id is for a row it owns, calls scrollIntoView and toggles a transient ring class (respecting prefers-reduced-motion — no pulse, just a static ring that fades).

Stories → board with the detail modal open

The story detail modal is opened today by local board state (setOpenStoryId in components/code/board.tsx). To open it from a different route, this spec makes the board URL-driven: it reads a ?story=<ref> search param and opens the matching story; closing the modal clears the param. This matches the codebase's "every view derives from the URL" philosophy (the shared ModuleRouter). 

// search → story selection
const href = `/code/${story.project_id}?story=${encodeURIComponent(story.ref)}`;
globalThis.history.pushState(null, '', href);
close();

// components/code/board.tsx — open from the URL
const storyRef = useSearchParams().get('story');
const openStory = storyRef ? stories.find((s) => s.ref === storyRef) : null;
If the selected story belongs to a different project than the one currently shown, the pushState to /code/[projectId] already routes the board to the right project before it reads ?story=, so the correct project's board renders with the modal open.

The global shortcut (annotated)

// components/shell/use-global-search-shortcut.ts
export function useGlobalSearchShortcut() {
  const { toggle } = useSearchActions();
  React.useEffect(() => {
    const onKey = (e: KeyboardEvent) => {
      // ⌘P (mac) / Ctrl P (win/linux). e.key is 'p' regardless of layout casing here.
      if ((e.metaKey || e.ctrlKey) && !e.shiftKey && !e.altKey && e.key.toLowerCase() === 'p') {
        e.preventDefault();   // claim it from the browser Print dialog
        toggle();
      }
    };
    globalThis.addEventListener('keydown', onKey);
    return () => globalThis.removeEventListener('keydown', onKey);
  }, [toggle]);
}
Implementation flags (resolve at build time, don't block the spec): confirm e.key handling across keyboard layouts; ensure the listener is attached once (in AppShell, a client component) and torn down; and verify the preventDefault() reliably suppresses Print in the target browsers. The react skill's "is this effect needed" guidance applies — a single window listener in the shell is the right home.

10 Keyboard & accessibility

KeyBehavior
⌘P / Ctrl PToggle palette (open when closed, close when open).
EscClose (Radix Dialog default); query cleared.
↑ / ↓Move the active result across both groups; wraps or stops at ends (pick one — stop-at-ends is simplest).
EnterSelect the active result and navigate.
typeUpdates query; active result resets to the first match.
  • On open, focus moves to the input automatically (Radix Dialog focus management).
  • Input is a combobox with aria-expanded; the results are a listbox; the active row is tracked via aria-activedescendant so screen readers announce it without moving DOM focus off the input.
  • Each result has role="option" and an accessible label including its kind ("Task" / "Code story") and where it lives.
  • Visible focus rings on all interactive elements (the repo's focus-visible:ring-accent-* tokens). Honor prefers-reduced-motion: the row-highlight on navigate is a static fade, no pulse.
  • Dark-theme tokens only (bg-surface, border-border, text-foreground, text-accent-teal); contrast meets the §5.4 quality floor.
  • Mobile: the header trigger is also reachable inside the hamburger; the palette is full-width with comfortable tap targets.

11 Testing & demo

Per CLAUDE.md, every behavioral change touches at least one test, and a user-facing change gets a demo doc once check is green.

  • Unit (jest): buildResults() — substring match on title/notes; story ref match (ALF-31); ranking order (title-prefix > title-substring > notes; ref-exact first); the 8-per-group cap and truncated counts; empty-query → no results.
  • Unit (jest): task-destination derivation (completed → /completed; folder → /folders/[id]; else inbox; subtask → ancestor view).
  • RTL: SearchPalette — opens on ⌘P and the keydown is preventDefaulted; typing filters; ↑↓ moves the active option (assert aria-activedescendant); Enter triggers navigation (mock pushState / the focus event); Esc closes and clears.
  • RTL: board.tsx opens the detail modal when ?story=<ref> is present and clears it on close.
  • Storybook + visual snapshot: the palette with mixed results (the storybook skill — capture diff, then approve the baseline).
  • Playwright (e2e): press ⌘P → type → select a task → land on its view with the row highlighted; press ⌘P → type a story title → select → Code board opens with the story modal.
  • Demo doc: docs/demos/alf-2-global-search/ — open the palette, screenshot results, show a task jump and a story jump; npm run demo -- verify before wrap-up.

12 Acceptance criteria

  • Pressing ⌘P (macOS) or Ctrl P (Windows/Linux) from anywhere in the app opens the search palette; the browser's Print dialog does not appear.
  • Pressing the shortcut again, Esc, or clicking the backdrop closes the palette and clears the query.
  • A discoverable "Search… ⌘P" trigger is visible in the top bar (and reachable on mobile via the hamburger); clicking it opens the palette.
  • Typing filters in real time across both tasks and code stories, results grouped under Tasks and Stories, each row badged by type.
  • Tasks match on title and notes; stories match on title, notes, and ref (e.g. ALF-31 finds that story).
  • Each group caps at 8 results with a visible "+N more" indicator when truncated (no silent cap).
  • ↑/↓ move the highlighted result; Enter opens it; the active option is exposed via aria-activedescendant.
  • Selecting a task navigates to its containing view (inbox / folder / completed) and scrolls the row into view with a brief, reduced-motion-aware highlight.
  • Selecting a story navigates to /code/[projectId]?story=<ref> and opens that story's detail modal; closing it clears the story param.
  • No new API/route/schema; the palette reads only the already-seeded stores. Search works offline-of-network (data already in memory).
  • Unit + RTL + Storybook snapshot + Playwright coverage per §11; root npm run check is green.

13 Out of scope / open questions

Out of scope (deliberately deferred)

Server-side search

No Postgres full-text / trigram / tsvector index. Client substring match over the in-memory stores is enough at hundreds of items; revisit if volume grows.

Fuzzy / typo tolerance

No fuzzy ranking library (Fuse.js et al.). Exact case-insensitive substring + the simple rank in §7.

Folders / projects / epics

The ticket scopes search to tasks and stories. Searching folder/project/epic names is not built here (see open question).

Knowledge items

item_type='knowledge' is reserved with no UI; nothing to search yet.

Recent / history

No persisted recent searches or "recently opened" seeding of the empty state. Possible follow-up.

Match highlighting

Bolding the matched substring inside each result is a nice-to-have, not required for acceptance.

Open questions (flag at implementation; defaults chosen)

  • Shortcut key. Default is ⌘P per the ticket, with a preventDefault() override of Print. Should we also bind ⌘K (the conventional palette key) as an alias, or switch entirely? Default: ⌘P only.
  • Completed tasks. useTasks() already holds completed items. Include them in results (badged/dimmed) or exclude by default? Default: include, visually de-emphasized.
  • Search scope vs. active module. Always search both modules (proposed), or scope to the active module with a toggle? Default: always both.
  • Folder/project names. Add folders/projects/epics as a third result group later? Out of scope now; noted for a follow-up ticket.
  • Empty-state seeding. Show a few most-recent items before typing, or just a hint? Default: hint only.
