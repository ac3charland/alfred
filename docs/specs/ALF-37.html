Refinement spec · Tasks module

ALF-37 — View all tasks by priority

Add a discrete priority level (High / Medium / Low) to tasks, settable from the existing task editor and shown as a badge on each row, and a new “By Priority” view that lists every top-level task ranked by priority with the due date as a tiebreaker. Spiritually analogous to the Code module’s cross-project Backlog, but ordered by a real priority field rather than manual swap.

1 · Context & problem

Today the Tasks module has three read views — Inbox (/), Folder (/folders/[id]) and Completed (/completed) — all routed client-side by components/tasks/task-views.tsx. Tasks are ordered structurally (parent → child tree) and otherwise by created_at. There is no notion of priority on a task: the items table (database/migrations/0001_initial_schema.sql) carries due_date, status, folder_id, parent_id and (since ALF-45) recurrence fields — but nothing that says “this matters more than that.”

The owner wants to triage across the whole system: see every task in one place, ordered by how important it is, regardless of which folder it lives in. The Code module already proves the “one flat, cross-cutting, ranked list” pattern with its Backlog (components/code/backlog.tsx, useBacklog() → stableSorted(…, byPriorityAsc)). ALF-37 brings that idea to tasks — but, per refinement, the ranking is a discrete priority level with due-date as the tiebreaker, set from the task editor, not a manually-swapped global rank.

Decisions locked during refinement
  • Priority = discrete levels (High / Medium / Low), not a manual swap rank.
  • Due date is the tiebreaker within a level (earliest first).
  • The list contains top-level tasks only (parentless); subtasks are not independently ranked.
  • Priority is set via a dropdown in the existing task editor menu (beside due date / notes), and shown as a badge on the row — no reorder UI.

2 · Proposed change (overview)

  1. Schema: a new Postgres enum task_priority = ('high','medium','low') and a nullable items.priority column (null = no priority). It rides through the task_items view automatically (the view is select i.*).
  2. Editor: a Priority dropdown added to TaskMetaPanel (components/tasks/task-row/task-meta-panel.tsx), alongside Due date, Repeat and Notes. Selecting a level (or “No priority”) calls the existing updateTask(id, { priority }) optimistic action.
  3. Badge: a new PriorityChip atom rendered on the task row (components/tasks/task-row.tsx), mirroring DueDateChip / RecurrenceChip. Shown for top-level tasks that have a priority set.
  4. View: a new By Priority view at /priority — a flat list of all top-level active tasks across Inbox and every folder, ranked by priority then due date, with a Show completed toggle (matching the Backlog). Derived entirely client-side from the already-seeded store via a new useTasksByPriority() hook (no new DB view, no extra fetch).
  5. Nav: a “Priority” link in the Tasks sidebar (components/tasks/folder-nav.tsx), routed by a new branch in TaskViews and a thin /priority page.
  6. API & validation: updateItemSchema accepts an optional nullable priority; the PATCH /api/items/[id] handler adds 'priority' to its allowed-update list.

3 · Data model & migration

One new migration — the next in sequence is database/migrations/0008_task_priority.sql (latest on disk is 0007_fix_swap_priority.sql). It follows the ALF-45 recurrence migration’s shape: enum + nullable column on items, plus an index for the sort path. We also extend the existing items_task_only_fields check so priority stays a task-only field. 

-- 0008_task_priority.sql — ALF-37: discrete task priority

-- ── Priority enum (declaration order IS the rank: high < medium < low) ──
create type task_priority as enum ('high', 'medium', 'low');

-- ── Add a nullable priority column to items (null = no priority set) ──
alter table items
  add column priority task_priority;

comment on column items.priority is
  'Discrete task priority (ALF-37). null = unprioritised. Task-only field; '
  'sorted high→medium→low with due_date as the tiebreaker in the By-Priority view.';

-- Index the sort/filter path.
create index items_priority_idx on items (priority);

-- ── Keep priority task-only, consistent with items_task_only_fields ──
alter table items drop constraint items_task_only_fields;
alter table items add constraint items_task_only_fields check (
  item_type = 'task'
  or (due_date is null and parent_id is null
      and status = 'active' and completed_at is null
      and priority is null)   -- ← added
);
Why no numeric rank / sequence? Unlike the Code Backlog (a unique bigint priority from a sequence, reordered by swap_code_priority()), ALF-37 ranks by a discrete enum. Many tasks share a level; the due date breaks ties. So there is no per-task unique rank, no swap RPC, and no reorder route — far less machinery than the Backlog.

Reads — nothing else to change

The Tasks/Inbox read path is the task_items view, defined as select i.* from items i where not exists (… code_items …) (database/migrations/0002_software_factory.sql). Because it is i.*, the new column flows through with no view edit. Subtasks load via the same flat getAllItems() fetch and are assembled client-side by buildTree, so priority is already present everywhere the store needs it. get_subtree() / complete_and_spawn() select explicit column lists and still compile unchanged — see Open questions for the one recurrence interaction worth a decision.

Regenerate types (part of the supervised step)

After the migration applies, regenerate frontend/lib/database.types.ts with supabase gen types (see the supabase skill). It is a generated file — commit the raw output verbatim, never hand-edit. The regenerated items Row gains: 

priority: Database["public"]["Enums"]["task_priority"] | null
// and a new Enums entry:  task_priority: "high" | "medium" | "low"

Item / ItemNode (lib/types.ts, lib/tree.ts) derive from that Row, so node.priority becomes available to the row and editor with no manual type work.

4 · Sort semantics

The By-Priority list sorts top-level active tasks by three keys, in order:

#KeyDirectionNull handling
1Priority levelHigh → Medium → Lownull (no priority) sorts last
2due_dateAscending (earliest / most overdue first)No due date sorts after dated tasks in the same level
3created_atAscending (stable final tiebreaker)—

A small rank map makes this explicit and testable. It lives with the other priority constants in a new frontend/lib/priority.ts: 

// lib/priority.ts
export type TaskPriority = 'high' | 'medium' | 'low';

// Lower rank = higher in the list. Unset (null) ranks last.
const RANK: Record<TaskPriority, number> = { high: 0, medium: 1, low: 2 };
export function priorityRank(p: TaskPriority | null): number {
  return p === null ? 3 : RANK[p];
}

export function byPriorityThenDue(a: Item, b: Item): number {
  const byLevel = priorityRank(a.priority) - priorityRank(b.priority);
  if (byLevel !== 0) return byLevel;
  const da = a.due_date ? Date.parse(a.due_date) : Number.POSITIVE_INFINITY;
  const db = b.due_date ? Date.parse(b.due_date) : Number.POSITIVE_INFINITY;
  if (da !== db) return da - db;
  return Date.parse(a.created_at) - Date.parse(b.created_at);
}

The view hook reuses the project’s stable sorter, exactly as useBacklog does (lib/sort.ts → stableSorted): 

// lib/stores/tasks-store.tsx — new derived hook
export function useTasksByPriority({ showCompleted }: { showCompleted: boolean }): Item[] {
  const items = useTasks();                      // flat seeded list
  return React.useMemo(() => {
    const topLevel = items.filter((i) => i.parent_id === null);   // top-level only
    const visible = showCompleted ? topLevel : topLevel.filter((i) => i.status === 'active');
    return stableSorted(visible, byPriorityThenDue);
  }, [items, showCompleted]);
}

6 · UI

6.1 · The By-Priority view

Same scaffold as the Backlog: a header (icon badge + serif title + one-line subtitle) with a Show completed ToggleButton, then a flat <ul> of rows grouped visually by level. Each row is a compact task line: title, the folder it lives in (or “Inbox”), a due-date chip when present, and the priority badge. Rows reuse the inline editor affordance so the owner can re-prioritise in place.

6.2 · The priority badge — PriorityChip

A new atom at components/atoms/priority-chip.tsx, modelled on RecurrenceChip / DueDateChip: a clickable Badge (components/atoms/badge.tsx) with a lucide icon and a level-mapped variant. The badge variants already exist — no badge changes needed:

6.3 · The editor control — Priority dropdown

Added to TaskMetaPanel (components/tasks/task-row/task-meta-panel.tsx) as a fourth field, beside Due date / Repeat / Notes. A simple select of the four options (No priority, High, Medium, Low) backed by the existing updateTask path — no draft/save dance needed since picking a value is a single atomic choice (unlike the free-text due-date/notes fields). Use the shadcn Select / DropdownMenu primitive already in the repo for consistency.

11 · Acceptance criteria

  • A task can be assigned a priority of High, Medium, or Low — or left unprioritised — from the task editor menu (beside due date / notes / repeat).
  • A task’s priority shows as a colour-coded badge on its row (High/Medium/Low); unprioritised tasks show no priority badge.
  • Setting/clearing priority is optimistic: the badge updates instantly and persists across reload; a failed save rolls back.
  • A By Priority view is reachable from the Tasks sidebar at /priority and deep-links / hard-loads correctly.
  • The view lists top-level (parentless) tasks only, across Inbox and every folder, in one flat list.
  • Order is High → Medium → Low → unprioritised; within a level, earlier due date first; tasks with no due date come after dated ones; created_at is the final tiebreak.
  • Completed tasks are hidden by default; a Show completed toggle reveals them (matching the Backlog).
  • priority is a nullable, task-only field (enforced by items_task_only_fields); non-task items keep it null.
  • PATCH /api/items/[id] validates and persists priority; invalid values are rejected.
  • All check:fast + check:slow gates pass; the new behaviour is covered by tests (§10) and captured in a demo doc.

12 · Out of scope & open questions

Out of scope

  • Manual within-level reordering (Backlog-style chevron swap / drag rank). Refinement chose discrete levels with a due-date tiebreaker; no per-task unique rank, swap RPC, or reorder route.
  • Independently ranking subtasks. The list is top-level only; subtasks inherit their parent’s context and are not surfaced as standalone ranked rows.
  • Priority on code / knowledge items. This is a task-only field; the Code Backlog keeps its own separate code_items.priority.
  • Filtering / grouping controls beyond the level ordering (e.g. “only High”, per-folder filters) — possible later, not now.
  • Auto-priority from due date / LLM classification — future, not in this ticket.

Open questions (for the implementation session / review)

  1. Recurrence inheritance. Should a spawned next occurrence inherit the recurring task’s priority? complete_and_spawn() inserts the new root with an explicit column list that does not include priority, so today it would reset to null. Recommend adding priority to that root insert so a recurring task keeps its level — small migration follow-up; flag if undesired.
  2. Badge colours / icons. Proposed High=destructive (red), Medium=alert (amber), Low=muted, with ChevronsUp/Equal/ChevronsDown. Confirm against the design language (§5.4) or adjust the mapping in PRIORITY_OPTIONS.
  3. Visual grouping vs. flat list. The mockup shows level group-headings (HIGH / MEDIUM / LOW). These are cosmetic; the ordering is what matters. Keep the headings, or render a single uninterrupted list? (Default: keep headings — easier to scan.)
  4. Editor control primitive. PrioritySelect as a shadcn Select vs. a DropdownMenu of PRIORITY_OPTIONS. Either fits; pick whichever already has a matching pattern in the repo at build time.
