Inbox multi-edit: select many items, classify / file / send-to-code in one pass

ALF-54 Add an explicit selection mode to the Inbox so the owner can act on several captured items at once — classify their type, file tasks into a folder, or send a batch into the Code module — instead of repeating the per-row "More actions" menu N times.

Context & problem

Capture is frictionless by design, so the Inbox fills with raw, unclassified items (plus any task still sitting at folder_id = null). Triage today is strictly one row at a time: each TaskRow carries a "More actions" dropdown (components/tasks/task-row/task-row-menu.tsx) that offers Classify as…, Move to…, Send to Code module…, and Delete — but only for that single item. Clearing an Inbox of a dozen captures means opening a dozen menus and making the same choice a dozen times.

There is no multi-select anywhere in the app yet. The only checkbox is CheckboxButton (the per-row completion toggle); the coordination stores that exist (ExpansionProvider, ActiveEditorProvider) manage expand/collapse and which row is being edited — neither tracks a selection set. So this ticket adds a genuinely new interaction layer, scoped to the Inbox where the repetition actually hurts.

Scope decided in refinement. The three bulk actions in v1 are Classify type, Move to folder, and Send to Code (project/epic). The selection model is an explicit "Select" mode toggle (persistent checkboxes, Done/Cancel to exit) — not always-on hover checkboxes. Bulk Delete was de-scoped during refinement and is listed under Out of scope.

How the Inbox renders today

The landing/Inbox screen is components/tasks/inbox-screen.tsx. Its list is a single <TaskList scope={{ type: 'inbox' }} /> derived client-side from the shared TasksProvider store — no per-view fetch. The header row already hosts an INBOX eyebrow and a CollapseAllButton; the new Select control lands right beside them.

Proposed change

1 · Selection mode — state machine

A new coordination store, InboxSelectionProvider, follows the same pattern as ExpansionProvider (transient, cross-row UI state that no single row owns — see the data-flow skill). It holds an active flag and a Set<string> of selected item ids, and is mounted inside the Tasks shell so the action bar and every Inbox row can read/write it.

Selection lifecycle. "Send to Code" inserts the gate dialog between Selecting · n and Applied; the other two actions apply immediately.

Store shape

// lib/stores/inbox-selection-store.tsx (new) — mirrors ExpansionProvider
interface InboxSelection {
  active: boolean;              // is select mode on?
  selectedIds: ReadonlySet<string>;
}
interface InboxSelectionActions {
  enter(): void;                   // Idle → Selecting·0
  exit(): void;                    // clear set + active=false (Done / Cancel)
  toggle(id: string): void;
  clear(): void;                   // empty the set, stay in mode
  isSelected(id: string): boolean;
}

The set is keyed by item id and only ever holds root Inbox items — the rows TaskList renders at scope.inbox. Subtask selection is out of scope. Entering or exiting the Tasks view, or an item leaving the Inbox, prunes stale ids.

2 · The Inbox row in select mode

When active is true, each Inbox TaskRow swaps its leading affordance (the drag handle / completion checkbox) for a selection checkbox, and the whole row toggles selection on click — inline editing, expand, and the per-row "More actions" menu are suppressed while selecting, so the row has exactly one meaning. A selected row gets the teal ring already used elsewhere for focus/active emphasis.

Action bar appears once ≥1 item is selected. Here the selection is all-unclassified, so Move to folder (task-only) is disabled while Classify and Send to Code are live.

3 · The bulk action bar & the enablement matrix

The bar is a new components/tasks/inbox-bulk-bar.tsx that renders only while active && selectedIds.size > 0. Each action's availability depends on the composition of the selection, keeping each bulk operation type-coherent (the same gating the single-row menu applies, lifted to a set):

Action Enabled when… Picker Underlying op (per item)
Classify as Task / Code every selected item is unclassified small dropdown (Task · Code) classifyItem(id, type)
Move to folder every selected item is task folder list (incl. "Inbox" to unfile) moveTask(id, folderId) — cascades subtree
Send to Code selection is non-empty (any Inbox item is eligible) shared gate dialog (project → epic) convertTaskToCode(item, projectId, epicId)

A disabled action shows a tooltip explaining the gate (e.g. "Only tasks can be filed into a folder"). The bar always shows the live count and a Done button (exits mode); pressing Esc also exits.

4 · Store: three new batch actions

The single-item actions already exist; v1 adds thin batch wrappers on TaskActions (and reuses code-store for the gate). The reducer already supports multi-id patches — dispatch({ type: 'patch', ids, patch }) takes an ids: string[] — so the optimistic step is a single dispatch over the whole set, and rollback restores the captured snapshot for all of them. No bulk HTTP endpoint is added: the wrappers fan the existing per-item routes out with Promise.allSettled and report partial failure. 

// lib/stores/tasks-store.tsx — new on TaskActions (signatures)
bulkClassify(ids: string[], itemType: 'task' | 'code'): Promise<void>;
bulkMove(ids: string[], folderId: string | null): Promise<void>;

// each follows the existing optimistic shape, but over the whole set:
async bulkClassify(ids, itemType) {
  const previous = tasksRef.current.filter((i) => ids.includes(i.id)); // snapshot
  await runOptimisticMutation({
    optimistic: () => dispatch({ type: 'patch', ids, patch: { item_type: itemType } }),
    apiCall: () => Promise.allSettled(ids.map((id) => api.updateItem(id, { item_type: itemType }))),
    reconcile: (rows) => dispatch({ type: 'upsert', items: rows }),
    rollback: () => dispatch({ type: 'upsert', items: previous }),
    onError: () => showToastRef.current("Couldn't classify some items"),
  });
}
Partial-failure policy. With allSettled, items that saved stay applied and items that failed roll back individually; the toast reports the count that failed (e.g. "2 of 5 couldn't be filed"). On success the store clears the selection and exits mode.

5 · Reusing the code gate for a batch

Per refinement, bulk send-to-code picks one project + epic for the whole selection. Rather than build a second dialog, the existing GateDialog (components/code/gate-dialog.tsx) is generalized from a single item: GateItem to items: GateItem[]: the project→epic picker is unchanged, the title/description pluralize ("Send 3 items to a project and epic"), and Confirm runs convertTaskToCode for each item, then removeGatedItem(id) for each (the items fall out of the task_items view once their code_items sidecar exists, exactly as in the single-item flow today). 

// components/code/gate-dialog.tsx — widen the contract; single-item callers pass [item]
interface GateDialogProperties {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  items: GateItem[];                       // was: item: GateItem
  onComplete: (stories: CodeStory[]) => void;  // was: (story) — now a batch
}

// Confirm handler (project + epic already chosen, button gated on both):
onSubmit: () => Promise.all(items.map((it) => convertTaskToCode(it, projectId, epicId))),
onSuccess: (stories) => { onComplete(stories); onOpenChange(false); },

The existing single-row "Send to Code module…" / "Convert to Code Story…" entry points keep working by passing a one-element array — a pure refactor covered by their current tests.

Acceptance criteria

  • The Inbox header shows a Select control; pressing it enters selection mode, replacing each Inbox row's leading control with a selection checkbox.
  • Clicking a row (or its checkbox) toggles its membership in the selection; the live count updates. Esc or Done exits mode and clears the selection.
  • The bulk action bar appears only when ≥1 item is selected and offers Classify as, Move to folder, and Send to Code….
  • Classify is enabled only when every selected item is unclassified, and sets item_type on all of them in one optimistic pass.
  • Move to folder is enabled only when every selected item is a task, and moves all of them (subtrees cascading) to the chosen folder or back to the Inbox.
  • Send to Code opens the gate, which picks one project + epic and admits every selected item to the factory; the gated items leave the Inbox.
  • A partial failure leaves saved items applied, rolls back only the failed ones, and toasts how many failed; full success clears the selection and exits mode.
  • Single-row "More actions" behavior (classify / move / send-to-code on one item) is unchanged.

Out of scope / open questions

  • Bulk delete — named in the ticket title but de-scoped in refinement; revisit as a fast follow once the selection model is in. (Single-row delete is unchanged.)
  • Selecting subtasks — only root Inbox rows are selectable in v1.
  • Bulk set priority / due date — the "etc." in the title; not in v1.
  • Multi-select outside the Inbox (folders, completed, the Code board) — this ticket is Inbox-only.
  • A server-side bulk endpoint — v1 fans the existing per-item routes out client-side; a batched endpoint is a possible later optimization, not required.
  • Select-all / range-select / shift-click — individual toggling only in v1; open question whether a "Select all" affordance is worth adding.
