ALF-56 — Make the webhook's alfred-block parsing position-independent

The ticket title (“Confirm real-time updates are listened to in all modules”) is a misnomer. The real defect is in the GitHub PR webhook Worker: it advances a ticket sometimes, and whether it does correlates with where the alfred block sits in the PR description. This spec pins the bug to a first-match regex and specifies a position-independent fix, kept in lockstep across the Worker and its enforcing CI check.

ALF-56 phase: refinement area: workers/ no UI

TL;DR

parseFrontmatter() (workers/src/frontmatter.ts) and the mirrored CI check (alfred-frontmatter.yml) both match the first ```alfred … ``` region in the PR body (/```alfred\s+([\s\S]*?)```/ — no g flag). When the operative block is at the top it is the first match and everything works. When it is at the bottom and any earlier ```alfred fence exists in the body (an example block — which our own refinement skill, the PR template, and spec prose all invite authors to paste), the parser locks onto the wrong block. The wrong block can still satisfy CI (green), while the Worker patches a non-existent ref → patchCodeItem returns 0 → a silent no-op → the ticket never advances. Fix: scan all blocks, keep only operative ones (valid phase + a real-shaped ref), and select the last one.

Context & problem

alfred's Software Factory has no LLM in the loop on the webhook side: a single Cloudflare Worker turns pull_request webhooks into deterministic code_items state transitions. The only signal it reads is a machine-readable alfred block carried in the PR description: 

```alfred
alfred-ticket: ALF-56
phase: refinement
spec-path: docs/specs/ALF-56.html
```

On each delivery the Worker verifies the HMAC, parses that block, plans a transition from (phase, action, merged), and PATCHes every ticket the block names (workers/src/index.ts). The on-screen swimlanes then update over the existing code_items realtime subscription — so a reliable parse is the first domino in the “live update” the title alludes to.

Why the title misleads. “Real-time updates in all modules” sounds like a frontend subscription gap. It is not. The frontend realtime path (code-store.tsx subscribing to code_items) works; what is flaky is the upstream webhook write that the subscription would relay. If the Worker never patches the row, no realtime event is ever emitted — so the symptom looks like “updates aren't listened to.”

The observed symptom

Tickets advance “sometimes.” The reporter noticed a correlation with the position of the alfred block in the PR description and named two real PRs:

PR Phase Block position in body Outcome
#136 · ALF-37 implementation near the top (right after the opening line) advanced ✓
#138 · ALF-50 implementation at the bottom (just above the Claude trailer) did not advance ✗

Root cause

Both the Worker parser and the CI check that mirrors it use a single, non-global, first-match regex to find the block: 

// workers/src/frontmatter.ts:34
const BLOCK_RE = /```alfred\s+([\s\S]*?)```/;   // ← no `g`, no `m`: matches the FIRST ```alfred … ``` only

// workers/src/frontmatter.ts:47
const block = BLOCK_RE.exec(body ?? '')?.[1];   // ← whatever the FIRST fence pairs to wins
# docs/code-module/repo-setup/alfred-frontmatter.yml:41 — the enforcing CI check, same regex
const m = b.match(/```alfred\s+([\s\S]*?)```/);
if (!m) { console.error("missing ```alfred block"); process.exit(1); }

BLOCK_RE.exec() on a regex without the g flag returns the first match. The field regexes (alfred-ticket, phase, spec-path) then run only on that first captured region. So the parser is hard-wired to read whichever ```alfred fence appears first in the body — never “the real one.”

Why “top works, bottom fails”

  • Block at the top: the operative block is, trivially, the first ```alfred fence in the body. First-match grabs it. Correct.
  • Block at the bottom: the operative block is the last fence. If anything earlier in the body is (or contains) a ```alfred fence, that earlier fence is matched first and the real block is never read. Wrong block — or no usable block.

An earlier ```alfred fence is not exotic — the project actively encourages it:

  • The refinement skill (.claude/skills/refinement/SKILL.md) shows the block format as a fenced example, and the launch prompt reproduces the fence verbatim.
  • A spec or PR narrative frequently quotes the block format to explain the change (this very spec does, above).
  • An implementation PR that touches the parser may paste a diff or code sample containing the literal fence.
Two regexes, one blind spot. The Worker and the CI check intentionally mirror each other “field-for-field so a PR that passes CI always parses here too” (frontmatter.ts header comment). They mirror the bug too: CI greenlights a body the Worker then mis-reads. Both must be fixed together or the contract drifts.

Honest investigation note

As they read today, the #136 and #138 bodies each contain exactly one ```alfred fence, so for those exact current bodies first-match alone would still land on the real block. That means one of: (a) a body was edited after merge (the webhook fires on the body at event time, which we cannot retrieve now), or (b) a second factor contributed (e.g. a missing code_items row for the ref, so even a correct parse PATCHes 0 rows). The first-match fragility is nonetheless a real, position-dependent defect that exactly produces “top works / bottom fails,” and the silent 0-rows no-op is a real observability gap. This spec therefore fixes the parser and adds the logging needed to diagnose the next occurrence conclusively. Confirming the live code_items state of ALF-50 is listed as an open question.

Proposed change

Replace first-match with scan-all-then-select, in both the Worker and the CI check. The selection is deterministic and position-independent:

  1. Find every ```alfred … ``` region (global, non-greedy) — not just the first.
  2. Keep only operative blocks: a block qualifies when it has a valid phase (refinement | implementation) and at least one ticket matching the real ref shape /^[A-Z][A-Z0-9]*-\d+$/. This filters out illustrative examples that use placeholders like <REF>.
  3. Select the last operative block. Operative blocks are rare-to-singular; when more than one survives the filter, the canonical machine block is the one authors append last (examples lead, the real block trails). Emit a warning log on ambiguity (>1 operative block).
  4. Zero operative blocks → undefined (the PR isn't ours), exactly as today.

Sketch — Worker parser

// workers/src/frontmatter.ts — illustrative, not final
const BLOCK_RE = /```alfred\s+([\s\S]*?)```/g;   // ← `g`: iterate ALL blocks
const REF_RE = /^[A-Z][A-Z0-9]*-\d+$/;          // a real ref, not a <REF> placeholder

export function parseFrontmatter(body?: string): AlfredFrontmatter | undefined {
  const operative: AlfredFrontmatter[] = [];
  for (const m of (body ?? '').matchAll(BLOCK_RE)) {
    const parsed = parseBlock(m[1]);              // the existing field-regex logic, factored out
    if (parsed && parsed.tickets.some((r) => REF_RE.test(r))) operative.push(parsed);
  }
  if (operative.length === 0) return undefined;
  if (operative.length > 1) console.warn(`alfred: ${operative.length} operative blocks; using the last`);
  return operative[operative.length - 1];      // last operative block wins
}

parseBlock() keeps the current field regexes verbatim (alfred-ticket:[ \t]*(.*), phase:[ \t]*(refinement|implementation), spec-path:[ \t]*(\S+)) — they are correct and well-tested; only block selection changes. The existing Stryker-disable comments move with the regexes.

Sketch — CI check (keep identical semantics)

# docs/code-module/repo-setup/alfred-frontmatter.yml — iterate all blocks, pick the last operative one
const blocks = [...b.matchAll(/```alfred\s+([\s\S]*?)```/g)].map(m => m[1]);
const ops = blocks.filter(blk => {
  const t = /alfred-ticket:[ \t]*(.+)/.exec(blk), p = /phase:[ \t]*(refinement|implementation)/.exec(blk);
  return t && p && t[1].split(',').some(r => /^[A-Z][A-Z0-9]*-\d+$/.test(r.trim()));
});
if (!ops.length) { console.error("missing/invalid ```alfred block"); process.exit(1); }
const blk = ops[ops.length - 1];   # then the existing phase / spec-path / archive checks run on blk
Guardrail. alfred-frontmatter.yml is a CI check the repo treats as scaffolding to copy into project repos, not tooling that gates this repo's commits — so editing it here is the deliberate, in-scope change, not a back-pressure bypass. The Worker fix is ordinary app code under TDD.

Observability (close the silent-failure gap)

Independent of selection, surface the two cases that are invisible today:

  • In handleWebhook (index.ts), when a parse succeeded but every ref matched 0 rows (matched.length === 0 with a non-empty frontmatter.tickets), console.warn the refs so a stuck ticket is greppable in Worker logs. The HTTP response stays 200 (a PR for another repo's ticket is still a benign no-op), but it is no longer silent.
  • When the body contains a ```alfred substring but yields no operative block, console.warn “malformed alfred block” rather than treating it as indistinguishable from a non-alfred PR.

Acceptance criteria

  • A PR body with an example ```alfred block (placeholder ref) above the operative block parses to the operative ticket — verified by a unit test that fails on main and passes after the change.
  • The operative block parses identically whether it sits at the top, middle, or bottom of the body (parametrized unit test).
  • A block whose only ticket is a placeholder (e.g. <REF>) is not selected; a body containing only such a block yields undefined.
  • With two operative blocks, the last is selected and a warning is logged.
  • Every existing frontmatter.test.ts case still passes (single-block bodies, comma lists, no-space-after-colon, missing-field undefined cases) — no regression.
  • When a parsed PR's refs all match 0 code_items rows, the Worker emits a console.warn naming the refs; the HTTP response stays 200.
  • alfred-frontmatter.yml uses the same scan + last-operative logic; a manual run over the example-above-real body validates the operative block (its ok: line names the real ref).
  • npm run check:fast -w workers is green (typecheck, lint, format, Jest). No --no-verify, no config weakened, no eslint-disable/@ts-ignore.

Test plan

Pure-logic change in a no-LLM Worker — the evidence is the Jest suite, not a UI demo, so no showboat demo doc is required (per the trivial/non-visual carve-out). Red-first:

  • Red: add parseFrontmatter tests for example-above-real and the position sweep; confirm they fail against the current first-match implementation.
  • Green: implement scan + filter + last-operative; all cases pass.
  • Mutation sanity (optional): the file carries Stryker-disable annotations; re-run npm run mutation -w workers if the survived-mutant baseline shifts, keeping disables justified (don't add new ones to dodge a survivor).

Out of scope / open questions

  • No frontend change. Per refinement decision, the task stores and the code_items realtime subscription are left exactly as-is — this ticket is the Worker parse only.
  • Selection rule — confirm last-vs-first. The spec picks last operative block (examples lead, real block trails). If the house convention is instead “the block is always first,” the rule flips to first operative. The validity filter makes both robust against placeholder examples; only the multi-real tiebreak differs. Flag for confirmation before build.
  • What actually froze ALF-50? Confirm the live code_items row + state for ALF-50 (and whether #138's body was edited post-merge). If the row never existed, that's a separate gap (gate/admission), and this fix only adds the log that would have surfaced it.
  • Body truncation. GitHub caps a PR body at 65,536 chars; a block beyond that cap is unparseable by anyone. Out of scope here (no evidence either PR approached the cap), but worth a one-line note in the skill that the block belongs in the body, not an over-long appendix.
  • Propagating the CI check. alfred-frontmatter.yml is a template copied into each consuming repo's .github/workflows/. Updating the template here does not update those copies; re-syncing them is a rollout follow-up, not part of this PR's diff.
  • Skill nudge (optional). Until this ships, authors can dodge the bug by keeping the operative block near the top and not pasting example blocks above it. Whether to add that interim note to the refinement skill is a judgment call, not a requirement.
