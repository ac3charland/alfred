Refinement spec · Tasks module

ALF-62 — Add project parsing to inbox task entry

Type ALF: ship dark mode in the Inbox capture box and the item is captured as a Code item, pre-assigned to the Alfred project, titled “Ship dark mode” — no menus, no gate, no epic required yet.

1 · Context & problem

Capture is meant to be frictionless (SPEC §1, §4). Today, turning a captured thought into a Code item is a two-step, menu-driven chore: capture it as an unclassified Inbox item, then open the row menu and pick Classify as → Code. Even after that, the item carries no project — the only way to attach a project is the Send to Code module gate, which also demands an epic.

The owner already thinks in project shorthand (“ALF: …”, “Alfred: …”). This ticket lets that shorthand do the classification work at the moment of capture: a recognized <project>: prefix classifies the item as Code and pre-assigns the project in one keystroke-cheap gesture, leaving the item in the Inbox for later triage.

The two “code” states in the data model (why this needs a new column)

There is a real distinction in the schema, and this feature lives in the gap between them:

State How it's represented Where it lives Project?
Code-classified inbox item items.item_type = 'code', no code_items row Inbox (kept by the task_items view) None today — this ticket adds it
Factory story items.item_type = 'code' + a code_items sidecar Code board (leaves the Inbox) code_items.project_id + epic_id (both NOT NULL)

Because a code-classified inbox item has nowhere to record a project, the prefix flow needs a lightweight, epic-free association on items itself. That is the crux of this spec.

2 · Resolved scope (from refinement Q&A)

  • Project link without an epic. The item stays in the Inbox as item_type = 'code' with a project assigned but no epic. A new nullable column, items.intended_project_id, holds that assignment.
  • Gate pre-population. When the item later goes through Send to Code module, the gate pre-selects the assigned project. In the bulk send path the project is pre-selected and locked when the whole selection shares one assigned project.
  • Surface. Parsing applies to the main Inbox capture box only (full mode, no folder, no parent). Folder captures, inline subtask captures, and the external POST /api/items (Siri) ingest path are untouched.

3 · Proposed change

3.1 · The prefix grammar

On capture, split the trimmed text at the first colon. The text before the colon (trimmed) is the candidate prefix; the text after (trimmed) is the remainder. The prefix matches a project when, compared case-insensitively, it equals that project's key (its “initials”, e.g. ALF) or its name (e.g. Alfred). A match triggers classification; anything else is captured verbatim as an unclassified item exactly as today.

Typed in the Inbox box Matched? Resulting item
ALF: add dark mode key ALF Code · project Alfred · title “Add dark mode”
alfred: add dark mode name (case-insensitive) Code · project Alfred · title “Add dark mode”
alf:fix bug (no space) key ALF Code · project Alfred · title “Fix bug”
ALF: rename the : separator key ALF (first colon only) Code · Alfred · title “Rename the : separator”
Note: buy milk (no such project) no match Unclassified · title “Note: buy milk” (verbatim)
buy milk (no colon) no match Unclassified · title “buy milk” (verbatim, unchanged)
ALF: / ALF:   (empty remainder) ignored Unclassified · verbatim (nothing worth capturing after the colon)

Matching precedence & ambiguity.

  • key is unique, so a key match is unambiguous and wins first.
  • Otherwise fall back to a case-insensitive name match. name is not unique in the schema — if the prefix matches the name of more than one project, treat it as no match (capture verbatim) rather than guessing.
  • Capitalization uppercases only the first character of the remainder (remainder[0].toUpperCase() + remainder.slice(1)); the rest is left as typed. A remainder that starts with a non-letter is left unchanged.
  • raw_capture preserves the original, unstripped text (e.g. "ALF: add dark mode") so nothing is lost for future re-processing (SPEC §3.2).

3.2 · Data flow

Matching needs the project list, which lives client-side in the CodeProvider store (it already wraps the Tasks view — see the gate dialog). So parsing happens client-side, at the Inbox capture box, before addTask.

3.3 · The pure parser new

Add frontend/lib/code/project-prefix.ts — a pure, dependency-free function so the matching rules are unit-tested in isolation (co-located project-prefix.test.ts). 

// frontend/lib/code/project-prefix.ts
import type { Project } from '@/lib/types';

export interface ProjectPrefixMatch {
  project: Project;   // the matched project
  title: string;      // remainder, prefix stripped + first letter capitalized
}

/**
 * If `text` starts with `<project name|key>:` (case-insensitive, first colon),
 * return the matched project and the cleaned title. Otherwise return null.
 *
 * - key match (unique) wins; else a case-insensitive name match, but only if
 *   exactly one project has that name (ambiguous name → null).
 * - empty remainder → null. No colon → null.
 */
export function parseProjectPrefix(
  text: string,
  projects: readonly Project[],
): ProjectPrefixMatch | null;

3.4 · Database migration

Add database/migrations/0012_intended_project.sql (next after 0011):

-- ALF-62 — a code-classified INBOX item may carry an intended project
-- (pre-factory, epic-free). code_items.project_id remains authoritative
-- once the item enters the factory; this is only the Inbox hint.
alter table items
  add column intended_project_id uuid references projects (id) on delete set null;

-- Only a code item may hold an intended project (keeps unclassified/task rows clean).
alter table items add constraint items_intended_project_code_only check (
  intended_project_id is null or item_type = 'code'
);

create index items_intended_project_id_idx on items (intended_project_id);
  • on delete set null: deleting a project must not orphan or break Inbox rows — the item simply loses its hint and stays a plain code item.
  • No new GRANTs / RLS changes. A column add inherits the table's existing grants and authenticated full access policy; migration-lint only flags missing sequence grants, and no sequence is created here.
  • The task_items view is select i.*, so it picks up the new column automatically. get_subtree lists columns explicitly but does not need it — subtasks are always task, never code.
  • Regenerate frontend/lib/database.types.ts with supabase gen types and commit it verbatim (it's a generated file — never hand-edit; see the supabase skill).
  • Optional tidy-up: enter_code_module may also clear intended_project_id (the sidecar's project_id supersedes it once in the factory). Harmless either way since the factory item leaves the task_items view; leave the RPC untouched unless trivial.

3.5 · API — create path

  • frontend/lib/api/schemas.ts — add intended_project_id: nullableUuid.optional() to createItemSchema (mirrors folder_id/parent_id). This flows into CreateItemInput automatically.
  • frontend/app/api/items/route.ts (POST) — pass intended_project_id: input.intended_project_id ?? null in the insert. Guard it the same way item_type is guarded: a row with a parent_id is forced to task, so it must never carry an intended project — set it to null whenever parent_id is present (belt-and-braces with the DB CHECK).

3.6 · Store — addTask

Extend AddTaskInput in frontend/lib/stores/tasks-store.tsx with two optional fields the Inbox capture box supplies when the prefix matched: 

interface AddTaskInput {
  text: string;                 // raw captured text → raw_capture (unchanged)
  folderId?: string | null;
  parentId?: string | null;
  // ALF-62 — supplied only by the Inbox capture box on a prefix match:
  title?: string;               // cleaned title (stripped + capitalized)
  itemType?: ItemType;          // 'code' on a match
  intendedProjectId?: string | null;
}

Inside addTask:

  • Build createInput with raw_capture: input.text, an explicit title: input.title ?? input.text, and item_type = parentId ? 'task' : (input.itemType ?? 'unclassified') (existing precedence, extended to honor an explicit type).
  • Pass intended_project_id only when a project is present and there's no parent.
  • makeOptimisticItem must reflect the cleaned title, item_type, and intended_project_id so the optimistic Inbox row renders as Code with its project immediately (then reconciles with the saved row).

3.7 · Capture box wiring (Inbox only)

CaptureBox is shared (Inbox, folders, subtasks), so parsing must be opt-in to keep it off the folder/subtask instances. Add a boolean prop parseProjectPrefix?: boolean (default false); the Inbox page's capture box passes true. When the prop is set (and thus only in full mode with no folderId/parentId):

  • Read the project list via useProjects() from the code-store.
  • On submit, run parseProjectPrefix(trimmed, projects). On a match, call addTask with { text: trimmed, itemType: 'code', title: match.title, intendedProjectId: match.project.id }; otherwise call addTask exactly as today.
  • The ghost/flourish, error handling, and out-typing behavior are unchanged.

3.8 · Gate pre-population

  • Extend GateItem (gate-dialog.tsx) with intendedProjectId: string | null; the single-row caller (task-row.tsx) and the bulk caller (inbox-bulk-bar.tsx) populate it from node.intended_project_id.
  • Single item: initialize the gate's projectId from items[0].intendedProjectId (still user-changeable). Epic selection is unchanged (user must still pick one).
  • Bulk: if every selected item shares the same non-null intendedProjectId, pre-select it and lock the project selector (render the project as a fixed, read-only chip instead of the interactive listbox); the user only picks the epic. If the selection's projects differ or any is null, fall back to today's interactive, unselected picker.

3.9 · Inbox affordance (project chip)

A code inbox row already shows a TypeBadge. When intended_project_id is set, also render the project's key as a small pill chip beside the badge (e.g. ALF) so the owner can see which project was assigned at a glance. Reuse the existing project-color helper (lib/code/project-color.ts) for the chip accent. Keep it modest — a read-only indicator, not an editor.

4 · Acceptance criteria

  • Typing ALF: add dark mode (key) or Alfred: add dark mode (name) in the Inbox capture box creates a Code Inbox item assigned to the Alfred project, titled “Add dark mode”, with raw_capture holding the original text.
  • Prefix matching is case-insensitive against both the project key and name.
  • The first letter of the post-colon remainder is capitalized; the rest is left as typed.
  • Only the first colon splits; later colons stay in the title.
  • A colon with an unrecognized prefix, an ambiguous (duplicate-name) prefix, no colon, or an empty remainder is captured verbatim as an unclassified item (no classification, no stripping).
  • The created Code inbox row shows the Code type badge and the assigned project's key chip.
  • Parsing does not run in folder capture boxes, inline subtask capture, or the POST /api/items Siri path — those behave exactly as before.
  • Opening Send to Code module… on such an item pre-selects its assigned project (epic still required).
  • Bulk-sending a selection that all shares one assigned project pre-selects and locks that project; a mixed/absent selection falls back to the interactive picker.
  • items.intended_project_id exists, is nullable, references projects(id) with on delete set null, and is constrained to code items; database.types.ts is regenerated.
  • The full npm run check passes green.

5 · Testing strategy (TDD)

  • Unit — project-prefix.test.ts: the full match matrix — key match, name match, case-insensitivity, no colon, no-such-project, ambiguous duplicate name, empty remainder, no-space colon, multiple colons, capitalization (letter, non-letter first char), surrounding whitespace.
  • RTL — capture box: the Inbox instance (prop on) maps ALF: foo to an addTask call with itemType: 'code', the project id, and title Foo; a plain string stays unclassified; a folder/subtask instance (prop off) never classifies.
  • Unit — store: addTask threads title, item_type, and intended_project_id into createItem and into the optimistic row.
  • API — route/schema test: createItemSchema accepts intended_project_id; POST /api/items inserts it and nulls it when a parent_id is present.
  • RTL — gate: single-item pre-selects from intendedProjectId; bulk locks on a unanimous project and stays interactive on a mixed selection.
  • E2E (Playwright), optional: capture ALF: … on the Inbox and assert the row renders as Code with the project chip and cleaned title.

7 · Out of scope / open questions

  • Out of scope: epic assignment or auto-creating a default epic (the item stays epic-free until the gate); parsing on the Siri/API ingest path; parsing in folder or subtask captures; fuzzy/partial project matching; LLM-based classification (SPEC §4.2 future layer); editing/adding a prefix after capture; a multi-project prefix syntax.
  • Confirm (reasonable default chosen): for a bulk send whose items have different assigned projects, this spec falls back to the interactive, unselected picker (no lock). If instead the bulk bar should refuse or split such a mix, say so in review.
