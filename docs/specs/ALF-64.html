ALF-64 — Account / instance switcher in top right

Give the top-right of the header a switcher that moves between two completely isolated alfred instances — a Personal second brain and a Work second brain — so tasks, code items, and everything else are physically separated, and work data never lands on a personal device.

Chosen approach: two deployments, not an in-app multi-account system. alfred is already a single-user app whose entire config comes from environment variables. The cleanest, lowest-effort, and most compliant way to isolate two second brains is to run the same codebase as two independent deployments (separate Supabase project + Vercel project each), and add a tiny top-right menu that links between them. This spec is an EPIC; it defines the goal, the deployment model, and the child stories to enter into alfred (§ Stories).

Context & problem

The owner keeps a personal "second brain" in alfred and wants a fully separate work one. The hard requirements:

  • Complete isolation of all data (tasks, folders, code items, captures) between the two brains.
  • Data-compliance: work tasks must not be logged into / reachable from the personal computer.
  • A switcher in the top right to move between them.

Today the header (frontend/components/shell/app-shell.tsx, lines 68–78) shows a single ghost Sign out button, wired to the signOut server action (frontend/lib/auth/actions.ts). Auth is a single manually-provisioned Supabase user; RLS is the deliberate single-user pattern (using(true), no user_id; product SPEC §7).

Why not an in-app multi-account switcher?

An in-app switcher (one deployment, multiple accounts, per-user RLS) was considered and rejected for this goal: it would put both brains in one Supabase database, and logging into the work account on the personal computer would fetch, render, and cache work data on that device — the exact compliance problem we're avoiding. It's also far more work (a user_id migration, an RLS rewrite, and a multi-session token roster). Two deployments give physical isolation with essentially no application code.

The deployment model

Two Vercel Projects, both connected to the same GitHub repo (both with Root Directory frontend/), each pointed at a separate Supabase project. This directly answers the two provisioning questions:

  • Can Vercel deploy multiple projects from one codebase? Yes — connecting the same repo to more than one Vercel Project is a first-class, supported pattern. The projects don't interfere; each has its own domain, Environment Variables, and build.
  • Do both URLs track the latest code? Yes — each project watches the same Production Branch (main), so a push to main triggers a production deploy on both projects. The code is identical across the two; only the per-project env vars differ. (Cost: two builds per push — fine on Hobby.)

"Switching" between them is just navigating to the other origin. They share no session, no cookies, and no database — which is exactly what makes the isolation airtight.

ALF-64c — Compliance hardening & docs optional · docs

  • Document the operating rule: install/bookmark the Work URL only on work devices, the Personal URL only on personal devices — the switch link is a convenience, not a bypass of that discipline.
  • Consider disabling offline/PWA caching of sensitive data if a service worker is ever added (none today).
  • Note the distinct INGEST_API_KEY per instance so a leaked personal key can't write to the work brain.

Defer if you want the minimum; ALF-64a + ALF-64b deliver the feature.

Acceptance criteria

ALF-64a — provisioning

  • Two live alfred URLs exist, each backed by its own Supabase project; data created in one is not present in the other.
  • Both projects deploy from main — a push updates both.
  • Each instance logs in independently with its own single user; each has its own INGEST_API_KEY and a working capture path.

ALF-64b — switcher UI

  • The top-right shows an accent instance pill (this instance's label) + chevron, replacing the bare Sign out button, on desktop and mobile.
  • Opening the menu shows the instance label and the signed-in email.
  • When NEXT_PUBLIC_OTHER_INSTANCE_URL is set, an Open <OtherLabel> link navigates to that URL; when unset, no link renders.
  • Sign out still signs out and redirects to /login (existing behavior preserved).
  • Optional accent theming reflects NEXT_PUBLIC_INSTANCE_ACCENT.
  • Menu is keyboard-navigable and screen-reader-labeled (Radix semantics); accent focus ring; prefers-reduced-motion respected.
  • getInstanceConfig() is unit-tested (all vars set; other-URL unset → link hidden; default accent). Storybook baselines include the menu in Personal (teal) and Work (amber) states. npm run check is green.

Testing & demo (ALF-64b)

Out of scope / open questions

  • No in-app account switching or per-user RLS. Isolation is achieved by separate deployments; the app stays single-user per instance and SPEC §7 is unchanged.
  • Auto-login on the other instance. Not attempted — the switch link lands you on the other origin, where you log in separately. Cross-origin session sharing is deliberately avoided (it would reintroduce the compliance risk).
  • Instance labels & domains ("Personal"/"Work", chosen URLs) are owner decisions made during ALF-64a provisioning; the code treats them as opaque env values.
  • More than two instances. The env model generalizes to a list, but v1 assumes exactly one "other". Extend later if a third brain is ever needed.
  • Cost. Two Vercel projects (two builds per push) and two Supabase projects. Expected to fit the free/Hobby tiers at this scale (SPEC §1 — hundreds of items).
  • Shared engineering. Both instances run identical code from main; a schema change must be migrated to both Supabase projects at deploy time — call this out in any future migration's rollout notes.
