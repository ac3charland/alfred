Refinement spec · phase: refinement

ALF-86 — Increase size of items/tasks on mobile

On a phone the task list reads small and cramped: 16 px checkboxes are hard to tap, the title is text-sm, and the date / priority / subtask-count badges crowd in from the right, squeezing long titles into three or four wrapped lines. This spec makes each top-level task a large, tappable card on mobile — with its subtasks nested inside the same card, not broken out into cards of their own.

Chosen design. Of the three options first floated on this ticket, the human picked “card per task”, with the refinement that a top-level item's whole subtree lives inside its card. This spec is now written to that single design.

Context & problem

Every item in the tasks module is rendered by a single recursive component, frontend/components/tasks/task-row.tsx, whose cosmetic classes are extracted into frontend/components/tasks/task-row.styles.ts. The list container (frontend/components/tasks/task-list.tsx) wraps the rows in one rounded card and separates them with hairline dividers (divide-y divide-border/50) — there is no gap between rows, and nested subtasks are just more divided rows inside that same shared container.

The row is not tuned for mobile today. The same geometry applies at every viewport:

Two consequences on a phone, both visible in the reference screenshot: (1) the 16 px checkbox is well below the ~44 px touch-target guideline, and (2) because the badges are right-aligned on the same line as the title, a long title like “Update monthly transfer amounts to account for daycare and new mortgage” wraps into four narrow lines while a wide empty gutter sits under the badges.

The app's mobile/desktop divide is the md breakpoint (sidebar hidden md:flex, mobile header md:hidden, and the ALF-88 actions rule). This design scopes the card treatment to below md; at md and up the desktop divide-y list and compact single-line rows are unchanged.

Proposed change

On mobile, replace the one-shared-container, hairline-divided list with a vertical stack of cards, one card per top-level (depth 0) item. Each card is a rounded surface panel with generous padding, a large checkbox + title in its head and a metadata footer (due date, priority, subtask count) below. Crucially, when a card is expanded, its entire subtask subtree renders inside that same card — indented, hairline-separated rows — rather than each subtask becoming its own card. The card boundary is drawn exactly once, at the top-level item, and encloses all of its descendants.

Anatomy

  • The card = one top-level item + its whole subtree. At depth 0 the item's outer wrapper gets the card chrome (surface bg, border, rounded-2xl, padding); the list places a gap between cards instead of dividers.
  • Card head — enlarged checkbox (≥ 24 px visual, ≥ 44 px hit area), text-base title, and the expand chevron (enlarged) when the item has children.
  • Card footer — the due / priority / subtask-count badges, moved below the title and aligned under it, so a long title takes the card's full width and no longer collides with the badges.
  • Nested subtasks (depth ≥ 1), inside the card — rendered in the existing collapsible subtree region, but within the parent card: indented via use-indentation.ts, separated by hairline dividers, each an enlarged compact row (checkbox + title + inline meta). Deeper levels keep indenting inside the same card. No nested card chrome.
  • Add-subtask box & “show completed” toggle stay inside the card (they already live in the subtree region), enlarged to match.

Where the card boundary is drawn

task-row.tsx renders one outer <li> per node, with its children recursively nested inside that same <li> (the subtree <ul> lives within the collapse wrapper). So the card chrome is applied to an inner wrapper of the depth 0 node that encloses both the row body and the subtree <ul> — that single seam is what puts subtasks inside the card automatically. Subtask nodes (depth ≥ 1) render as today's rows (enlarged), never as cards. The card chrome, footer restructure, and gapped list are all gated behind the default (< md) styles with md: prefixes restoring today's compact desktop layout — the same responsive convention as rowActionsClass. No useMediaQuery/JS branching; pure Tailwind responsive classes keyed on depth === 0.

Implementation notes

  • Enlarge the real tap target, not just the glyph. The checkbox (frontend/components/atoms/checkbox-button.tsx) and the expand chevron (IconButton) must present a ≥ 44 px hit area on mobile even if the drawn box is ~24–26 px.
  • Keep the badge components; change only their size/placement context. The due pill, priority glyph, subtask count and type badge come from shared atoms (badge.tsx, priority-chip.tsx, due-date-chip.tsx, type-badge.tsx). The parent's footer and the subtask rows reuse those atoms; prefer bumping sizes through them / the row's wrappers over new one-off values (the inline text-[13px] px-3 subtask-count pill in task-row.tsx should move to a named class alongside the others).
  • Indentation still works. Nested subtasks keep use-indentation.ts (depth*1.25 + 0.75rem); verify the offsets read cleanly inside the card's padding, so the first subtask level sits just inside the parent's title column rather than flush to the card edge.
  • Collapse animation. The subtree still expands/collapses via AnimatedHeightCollapse; it now animates within the card, so the card grows/shrinks with it.
  • Chrome classes are locked by a test. New responsive/card classes are added to the *Class constants in task-row.styles.ts (and any new list-container class), asserted in task-row.styles.test.ts — not sprinkled inline.

Acceptance criteria

  • At a mobile viewport (< md), each top-level task renders as a distinct card — rounded surface panel with a border and a visible gap between cards (no shared divide-y container).
  • A top-level card's checkbox presents a ≥ 44 px tap target with a ≥ 24 px visual box; the title renders at text-base (16 px) or larger.
  • The due-date, priority, and subtask-count badges render in a footer below the title (not to its right), so a long title takes the card's full width without colliding with them.
  • When a card is expanded, its subtasks render inside that same card — indented, hairline-separated rows — and not as separate cards. Deeper nesting stays within the top-level card, progressively indented and legible.
  • Expand/collapse animates within the card (the card grows/shrinks); the add-subtask box and “show completed” toggle appear inside the card.
  • At md and above, the list and row layout are unchanged from today (desktop density preserved).
  • Complete, inline-edit, drag, and reparent interactions still work at the larger sizes.
  • New card/row classes are asserted in task-row.styles.test.ts; a mobile-viewport Storybook snapshot covers a card with an expanded nested subtree; npm run check is green.
  • A demo doc under docs/demos/<branch>/ captures the before/after at a mobile viewport (per the showboat skill) and npm run demo -- verify passes.

Out of scope / open questions

  • Exact mobile breakpoint. Assumed md (the app's existing mobile/desktop divide). Flag if a different threshold is wanted.
  • Subtask meta layout. Inside the card, subtask rows show their badges inline (compact) rather than each getting a parent-style footer, to keep card height bounded; the exact inline-vs-stacked treatment for a subtask with many badges is an implementation detail to settle during build.
  • Folder headers, the capture bar, and other tasks-module chrome are unchanged — this ticket is the task row / list only.
  • No changes to desktop density, information shown, or task behaviour — sizing/layout only.
