ALF-86 — Increase size of items/tasks on mobile

On a phone the task list reads small and cramped: 16 px checkboxes are hard to tap, the title is text-sm, and the date / priority / subtask-count badges crowd in from the right, squeezing long titles into three or four wrapped lines. This spec proposes three redesigns of the task row that make items larger and more legible on mobile. Pick one and this spec will be rewritten to that single option with finalised acceptance criteria.

Decision needed: choose Option A (scale up in place), Option B (metadata on its own line — recommended), or Option C (card per task). Everything below the “Options” section (files, testing, acceptance) is written to be true of whichever option is chosen and will be narrowed once you decide.

Context & problem

Every item in the tasks module is rendered by a single recursive component, frontend/components/tasks/task-row.tsx, whose cosmetic classes are extracted into frontend/components/tasks/task-row.styles.ts. The list container (frontend/components/tasks/task-list.tsx) wraps the rows in a rounded card and separates them with hairline dividers (divide-y divide-border/50) — there is no gap between rows.

The row is not tuned for mobile today. The same geometry applies at every viewport:

Two consequences on a phone, both visible in the reference screenshot: (1) the 16 px checkbox is well below the ~44 px touch-target guideline, and (2) because the badges are right-aligned on the same line as the title, a long title like “Update monthly transfer amounts to account for daycare and new mortgage” is forced to wrap into four narrow lines while a wide empty gutter sits under the badges.

The app's mobile/desktop divide is the md breakpoint (sidebar hidden md:flex, mobile header md:hidden, and the ALF-88 actions rule). Every option here scopes its enlargements to below md so the desktop density is unchanged, matching that established convention.

Options

The three options are points on a spectrum from “same layout, bigger” to “new container per task”. All three: enlarge the checkbox to a ≥ 24 px visual target, bump the title to text-base (16 px), enlarge the badges, and grow the expand chevron — the difference is where the metadata goes and how rows are separated. Mockups are rendered at a phone width so you can compare directly against the screenshot.

Option A — Scale up in place

Keep the exact single-line information architecture (checkbox · title · right-aligned badge cluster) and simply scale everything up on mobile: a 24 px checkbox with a ≥ 44 px hit area, text-base title, larger pills, larger chevron, more row padding (py-3). Smallest change; keeps desktop and mobile structurally identical.

Option A — same layout, bigger controls & text.

Pros
  • Smallest, lowest-risk diff — mostly size tokens.
  • Mobile and desktop stay structurally identical (one mental model).
  • Preserves current density — most tasks per screen of the three.
Cons
  • Does not fix the core wrapping problem: bigger badges crowd the title even more, so long titles still wrap to 3–4 lines.
  • Bigger text with the same layout can feel more cramped, not less.

Option B — Metadata on its own line (recommended)

On mobile, give the title the full row width and move the badges (due date, priority, subtask count, type) down to a dedicated meta strip below the title. Checkbox and title enlarge as in A, but because nothing competes for the title's horizontal space, long titles wrap far less and the metadata reads as a tidy chip row — the pattern of most native mobile task apps. Desktop keeps today's single-line layout; the stacked meta strip is a < md-only rearrangement.

  • A meta-less task (no date/priority/subtasks) shows only the title row — must collapse the empty strip cleanly.

Option C — Card per task

Replace the divided list with individually-bordered, spaced cards on mobile: each task is a rounded surface panel with generous padding, a large checkbox and title on top and a metadata footer below. The biggest, most tappable, most visually distinct treatment — closest to a “widget” list.

Shared implementation notes (any option)

  • Scope to mobile via the md breakpoint. Enlargements live behind the default (< md) styles with md: prefixes restoring today's compact desktop values — mirroring the existing rowActionsClass convention. No useMediaQuery/JS branching; pure Tailwind responsive classes.
  • Enlarge the real tap target, not just the glyph. The checkbox (CheckboxButton, frontend/components/atoms/checkbox-button.tsx) and the expand chevron (IconButton) must present a ≥ 44 px hit area on mobile even if the drawn box is 24 px, so the whole item is comfortably tappable.
  • Keep the badge components; change only their size context. The due pill, priority glyph, subtask count and type badge come from shared atoms (badge.tsx, priority-chip.tsx, due-date-chip.tsx, type-badge.tsx). Prefer bumping sizes through those atoms / the row's wrappers over hard-coding new one-off values (the inline text-[13px] px-3 subtask-count pill in task-row.tsx should move to a named class alongside the others).
  • Indentation still works. Nested subtasks use use-indentation.ts (depth*1.25 + 0.75rem); whatever option is chosen must keep subtask indentation legible at the larger sizes (esp. Option C's cards).
  • Chrome classes are locked by a test. task-row.styles.test.ts asserts the row's className constants; new responsive classes are added there (and to *Class constants in task-row.styles.ts), not sprinkled inline.

Acceptance criteria

Finalised once an option is chosen; these hold for whichever is picked:

  • At a mobile viewport (< md), the checkbox presents a ≥ 44 px tap target with a ≥ 24 px visual box.
  • At a mobile viewport the task title renders at text-base (16 px) or larger.
  • At md and above, the row's size/layout is unchanged from today (desktop density preserved).
  • The due-date pill, priority glyph, subtask-count pill and type badge remain present, legible, and correctly coloured at the new sizes.
  • A long title no longer collides with the badge cluster (Options B/C: title takes full width; Option A: badges still fit without clipping).
  • The expand/collapse chevron, inline edit, complete, and drag interactions still work at the larger sizes; nested subtasks stay legibly indented.
  • New responsive classes are asserted in task-row.styles.test.ts; a mobile-viewport Storybook snapshot covers the layout; npm run check is green.
  • A demo doc under docs/demos/<branch>/ captures the before/after at a mobile viewport (per the showboat skill) and npm run demo -- verify passes.

Out of scope / open questions

  • Which option — the one decision to make; the spec is then rewritten to just that option.
  • Exact mobile breakpoint. Assumed md (the app's existing mobile/desktop divide). Flag if a different threshold is wanted.
  • Folder headers, the capture bar, and other tasks-module chrome are unchanged — this ticket is the task row only.
  • No changes to desktop density, information shown, or task behaviour — sizing/layout only.
