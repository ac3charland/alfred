ALF-117 — Add ability to reorder subtasks

Refinement Tasks module DnD · DB · store Spec only — no implementation. Build in a later session.

1 · Context & problem

Subtasks in alfred are an arbitrary-depth adjacency tree (items.parent_id). Today a parent's children have no user-controllable order: buildTree's sortForest sorts every subtask group by created_at ascending (oldest-first, chronological), and the Folder view additionally re-ranks every level by priority → due → created_at via sortNodesByPriority.

When you decompose a task into steps, you want to arrange those steps into the order you'll do them. ALF-117 keeps the current default — a subtask group is shown in the order the subtasks were created — and lets you override it with a manual sort by dragging. Once applied, the manual order persists across reloads and shows the same in every view; priority stops reordering subtasks and becomes a display signal only.

2 · Decisions (locked during refinement)

Scope
Subtasks only. Any row with a parent_id (at any depth) becomes manually orderable among its siblings. Top-level (root) order is unchanged — Inbox stays newest-first, the Folder view stays priority-ranked. Roots get no reorder affordance.
Gesture
Drop in the gap between siblings. Thin, layout-neutral drop strips sit at each subtask boundary. Dropping onto a row body keeps its current meaning (re-parent); dropping in a gap positions the subtask at that slot. Reuses the promote-zone + fractional-rank patterns already in the repo.
Order model
Creation order by default; a manual sort overrides it. A non-primary (subtask) row is displayed in the order it was created — in every active view — until the user drags it; from then on that group keeps the manual order, and new subtasks append at the bottom. Priority never reorders subtasks: it stays a signal only — row chips, folder badge counts, and the dedicated By-Priority view are all unchanged. (Roots are still priority-ranked in the Folder view; only subtasks drop the priority sort.)
One storage, two states, no flag. The fractional sort_order column (§4.1) is seeded from created_at, so "creation order" and "manual order" are the same total order per group — the default simply is creation order, and a drag is the only thing that diverges it. No separate "has-been-sorted" boolean is needed, and because created_at is still on the row, a future "reset to creation order" action stays cheap (see §5).

4 · Proposed change

4.1 · The fractional-rank column

Add one ordering column to items, following the exact pattern already proven for code-story Backlog ranking (migrations 0005 → 0014): a fractional rank so a single reorder is one UPDATE at the midpoint of two neighbours — never a renumber of the list.

database/migrations/0015_subtask_sort_order.sql (new) 
-- ALF-117: manual subtask ordering. Fractional rank (double precision) so a reorder
-- inserts at the midpoint of two neighbours — one row UPDATE, no renumber (as ALF-110
-- did for code_items.priority). Lower = earlier (top of the sibling group).

create sequence item_sort_order_seq;

alter table items
  add column sort_order double precision not null default nextval('item_sort_order_seq');

comment on column items.sort_order is
  'Manual sibling order for subtasks (ALF-117). Lower = earlier. Only meaningful WITHIN a '
  'parent_id group; roots ignore it (Inbox=created_at desc, Folder=priority). Fractional: a '
  'reorder inserts at the midpoint of two neighbours so no sibling is renumbered.';

-- Preserve today's order: seed from created_at so existing subtasks keep their chronological
-- (oldest-first) sequence. Only the RELATIVE order within each parent matters, so a global
-- row_number is sufficient.
with ranked as (
  select id, row_number() over (order by created_at) as rn from items
)
update items i set sort_order = ranked.rn from ranked where ranked.id = i.id;

-- Park the sequence above every backfilled value so fresh inserts land at the bottom.
select setval('item_sort_order_seq',
              coalesce((select max(sort_order) from items), 0)::bigint + 1, false);

-- security_invoker inserts run the column default's nextval() as `authenticated`, which
-- needs USAGE on the sequence (see 0008 — the bug code_priority_seq hit). Grant it.
grant usage on sequence item_sort_order_seq to anon, authenticated, service_role;

-- task_items is `select i.*`; PostgreSQL freezes the column list at CREATE, so recreate the
-- view to re-expand it and surface sort_order (the 0011 freeze gotcha).
create or replace view task_items with (security_invoker = true) as
  select i.* from items i
  where not exists (select 1 from code_items c where c.item_id = i.id);

No global unique index (unlike code_items.priority): ordering is scoped per parent, so global collisions are harmless and no atomic-swap RPC is needed. A brand-new subtask gets the next (largest) sequence value → it appends at the bottom of its sibling group, matching today's created-order default with zero route changes.

Generated types. Item is derived from frontend/lib/database.types.ts. After the migration, regenerate that file (supabase gen types) and commit the raw output — never hand-edit it (see CLAUDE.md “Generated files”). TypeScript will then flag every Item literal (factories, fixtures, stories) that must add sort_order.

4.2 · Sort layer — make subtasks honour sort_order

Two pure changes in the read layer; roots are deliberately left alone.

FileChange
frontend/lib/tree.ts
buildTree / sortForest		 Roots keep created_at desc (unchanged). Sort each children group by sort_order ascending instead of created_at. (Restructure sortForest so the root call and the recursive child calls use different comparators.)
frontend/lib/priority.ts
sortNodesByPriority		 Rank only the top-level nodes it's handed; stop re-sorting children by priority — preserve the sort_order order buildTree already applied. (Model C: the Folder view no longer reorders subtasks by priority.) The By-Priority view and folder badge counts are untouched — they read priority, not child order.

useTasksByPriority and the Completed view already derive subtrees from buildTree, so they inherit the new sort_order child ordering for free.

new action — tasks-store.tsx 
reorderSubtask(id, { parentId, sortOrder }): Promise<void>
  // Optimistic: patch { sort_order: sortOrder } on `id`.
  //   If parentId differs from the row's current parent_id, ALSO set parent_id (+ adopt the
  //   new parent's folder_id, cascading folder_id to the subtree) — i.e. reuse reparentTask's
  //   folder-cascade + cycle guard. Same parent → pure { sort_order } patch, no descendant writes.
  // apiCall: updateItem(id, { sort_order, ...(parent changed && { parent_id, folder_id }) })
  //          + per-descendant folder PATCHes when the folder changed (as reparentTask does).
  // reconcile: upsert saved rows.  rollback: upsert the captured subtree.  onError: toast "Couldn't move task".

The cross-parent case is a natural consequence of gaps existing under every expanded parent: dropping subtask A into a gap under a different parent B both re-parents A into B and places it at that exact slot. It shares reparentTask's cycle guard (a subtask may never land in a gap among its own descendants) and folder cascade. Consider factoring the shared cascade helper rather than duplicating it.

4.4 · DnD wiring — gap strips + resolver

A new pure module mirrors lib/dnd/reparent.ts and promote-to-root.ts, plus a thin gap-strip component rendered inside each subtask <ul>.

Row body = re-parent target (unchanged). Thin gap strips = reorder targets; the drop lands at the midpoint of its neighbours' sort_order.

lib/dnd/reorder-subtask.ts (new) — pure & unit-tested 
export const REORDER_GAP_PREFIX = '__reorder-gap__';

// Encode which slot a gap represents. The handler reads neighbour sort_order values from the
// store by id, so the id only has to locate the slot: parent + the insertion index.
export function reorderGapId(parentId: string, index: number): string { ... }
export function isReorderGap(overId: string | null): boolean { ... }

// The fractional midpoint — THE unit-test target (jsdom can't drive a real drag).
export function computeInsertOrder(prev: number | null, next: number | null): number {
  if (prev === null && next === null) return 0;   // empty group
  if (prev === null) return next - 1;              // top gap
  if (next === null) return prev + 1;              // bottom gap
  return (prev + next) / 2;                          // between
}

// Turn (dragged id, gap slot, the parent's ordered siblings) into the reorder to run, or null.
// orderedSiblings EXCLUDES the dragged row. Guards → null:
//   • not a gap; • dropping back into the dragged row's own current slot (no-op);
//   • cross-parent into the dragged row's own subtree (cycle — caller supplies subtreeIds).
export function resolveReorder({ draggedId, gapParentId, orderedSiblings, insertIndex,
  draggedParentId, subtreeIds }): { itemId, parentId, sortOrder } | null { ... }

Component changes

  • task-list.tsx / task-row.tsx: render a SubtaskGap droppable at the top of, between, and at the bottom of each rendered active subtask group (one more gap than rows). Strips are layout-neutral — zero flow height, positioned over each boundary (absolute / negative-margin) so revealing the teal insertion line never reflows the list (the reflow pitfall the dnd-kit skill calls out). A strip only lights up while a subtask is being dragged.
  • task-dnd-provider.tsx: in handleDragEnd, branch on isReorderGap(overId) before the folder / promote / reparent branches; resolve via resolveReorder and call reorderSubtask. The collision function should test reorder-gap strips before row bodies so a boundary hover reliably reads as reorder rather than re-parent.
  • Keyboard / a11y — TaskRowMenu: add “Move up” and “Move down” items on active subtask rows (hidden on roots, and on the first/last sibling respectively). Each computes computeInsertOrder against the sibling it swaps past and calls reorderSubtask. This is the deterministic, screen-reader-friendly path (the gap strips aren't in a SortableContext, so keyboard DnD across them is unreliable) — and it's the stable handle for E2E.

5 · Out of scope / open questions

  • Reordering top-level tasks — roots keep automatic order (decided). If wanted later, it's a separate ticket that must decide how manual order coexists with newest-first / priority-rank roots.
  • Reordering completed subtasks — the Completed view renders in sort_order but offers no reorder gesture.
  • “Reset to creation order” — since created_at is still on every row, a per-parent action that re-seeds sort_order from created_at (undoing a manual sort) is cheap to add later. Not in this ticket unless you want it — flag during implementation.
  • Fractional-precision rebalance — double precision exhausts after ~50 consecutive midpoint halvings of the same interval. Matching the existing code_items.priority approach, no periodic re-spacing job is included; acceptable at single-user scale. Note it if it ever bites.
  • Recurring-task reset — computeOptimisticSpawn copies a root (which ignores sort_order); the server owns the reset subtree's ordering on the next load. No special handling needed, but the implementer should confirm the reset subtree reads back in a sensible order.

6 · Acceptance criteria

  • Migration 0015 adds items.sort_order (double precision, not-null, sequence default), backfills from created_at, grants sequence USAGE, and recreates task_items; database.types.ts is regenerated (not hand-edited).
  • Existing subtasks render in the same order as before the migration (backfill preserves chronological order); no visible change until a user reorders.
  • Dragging a subtask into a gap between two siblings moves it there; the new order persists across reload and shows identically in Inbox, Folder, and By-Priority.
  • In the Folder view, sibling subtasks follow creation order by default and the manual order once sorted — never priority; root rows there are still priority-ranked, priority chips still render, and folder badge counts are unchanged.
  • Dropping a subtask onto a row body still re-parents (existing behaviour intact); dropping on a list edge still promotes to root.
  • A new subtask appends at the bottom of its parent's current children.
  • Dropping a subtask into a gap under a different parent re-parents and positions it at that slot; a drop into the dragged row's own subtree is a no-op (no cycle).
  • “Move up” / “Move down” menu items reorder an active subtask by one slot and are hidden on roots and at the ends of a sibling group.
  • Gap strips are invisible until a subtask drag and never reflow the list when revealed; reduced-motion is respected.
  • A failed reorder PATCH rolls the row(s) back and toasts, like every other optimistic mutation.
