ALF-3 — ⌘K to navigate (command palette)

Add a keyboard-first navigation palette: pressing ⌘K (⌘K on macOS, Ctrl K on Windows/Linux) pops a centered command palette listing every place you can go — the two modules, the cross-cutting views, every folder, and every project — filterable by typing, driven entirely by the keyboard. Pick one and it jumps you there instantly.

Context & problem

Navigating alfred today means reaching for the sidebar with the mouse: the Tasks ⇄ Code switcher (components/shell/view-switcher.tsx), the folder list (components/tasks/folder-nav.tsx), the project list (components/code/project-nav.tsx), and the cross-cutting Priority / Completed / Backlog links. As the number of folders and projects grows, hunting the sidebar for one of them is slow, and on the Code module the Tasks folders aren't even visible (the sidebar swaps per module), so jumping from a folder to a project is a two-step mouse dance.

The app already ships a global content search bound to ⌘P (components/shell/search-box.tsx + search-results.ts): it fuzzy-finds individual task and story rows by title/notes/ref and jumps to the one you pick. That answers "where is this item?". It deliberately does not answer "take me to that place" — an empty query shows nothing, folders and projects are only used as subtitles (never as destinations), and the modules/views aren't in it at all.

Scope decision (confirmed with owner). ⌘K is a destination navigator, a separate concern from ⌘P content search. ⌘P stays exactly as it is. The two coexist: ⌘P = "find an item", ⌘K = "go to a place". This mirrors the split in tools like Linear (go-to vs. search) and keeps each surface simple.

Proposed change

Introduce a command palette mounted once in the shell. It is a centered modal overlay (not the anchored header dropdown that ⌘P uses) built on the existing FormDialog atom, containing a filter input over a grouped, keyboard- navigable list of navigation destinations. Selecting a destination performs the same client-side history.pushState switch the sidebar links already use (the ViewLink convention), then closes the palette.

The destinations

Destinations come from two sources — a small static set (modules + views, known at build time) and the dynamic folders and projects read from the already-seeded stores (useFolders(), useProjects()). No network: the shell layout already fetches and seeds both (app/(shell)/layout.tsx).

Filtering & ranking

Extract a pure, DOM-free module command-destinations.ts — the direct sibling of search-results.ts — so the matching rules are exhaustively unit-testable on their own. It takes the query plus the folder/project snapshots and returns the grouped, ranked, capped destination list.

  • Empty query lists everything. Unlike ⌘P (empty → nothing), an empty query shows the full destination set grouped as above — the palette doubles as a browsable "where can I go?" menu.
  • Match is case-insensitive on the destination label (folders/projects match on their name; projects additionally match on their key, so typing a ref prefix like ALF surfaces that project). Rank reuses the search-results.ts ladder: label-prefix (0) beats label-substring (1); ties break by the group's natural order (static order for Go-to, the store's existing order for folders/projects). Non-matches are dropped.
  • Grouping is preserved in the output (Go to → Folders → Projects), and the groups are flattened into one ordered array for ↑/↓ keyboard traversal, mirroring flattenResults. Each destination carries a stable id for aria-activedescendant.
  • No per-group cap needed at MVP scale (hundreds of folders/projects at most). If a future cap is wanted it follows the RESULTS_PER_GROUP "+N more" pattern — noted, not built.

Opening, keyboard model & selection

  • Open: a single global keydown listener claims ⌘K/Ctrl K (bare, no Shift/Alt), preventDefault()s to take it from the browser's default (Firefox/Chrome bind it to the address/search bar), and toggles the palette open. This is a new hook use-command-palette-shortcut.ts, a sibling of the existing use-global-search-shortcut.ts — same shape, different key and a toggle instead of a focus callback. It fires globally, including while another input is focused (a nav palette should always be reachable).
  • Focus & typing: on open the filter input autofocuses (Radix Dialog traps focus); the query resets to empty each open, so every invocation starts from the full list.
  • Keyboard: ↑/↓ move the active row (clamped to the flattened list), ↵ selects the active row, Esc closes (Radix Dialog's own Esc/overlay-click also closes). The active row follows the mouse on hover so pointer and keyboard agree — the exact interaction contract already implemented in search-results-popover.tsx/search-box.tsx, reused here.
  • Select = history.pushState(null, '', href) then close the palette. Every view re-derives from the URL via ModuleRouter, so no reload and no RSC round-trip. Selecting the destination you're already on is a harmless no-op push that just closes the palette.

Where it mounts & discoverability

  • Mount <CommandPalette /> once inside AppShell (components/shell/app-shell.tsx), alongside ToastViewport. It reads the already-mounted FoldersProvider/CodeProvider stores; no new provider is required. The palette owns its own open state locally (nothing else needs to read it), so — unlike SearchProvider, which exists only to sync two input fields — no store is added.
  • Discoverability (in scope, minimal): the desktop ⌘P field already renders a <kbd>⌘P</kbd> hint. Add an equally quiet ⌘K affordance so the shortcut is discoverable without a mouse-only user ever knowing it exists — e.g. a small ⌘K hint in the sidebar. Exact placement is an implementation detail; a hint must exist somewhere in the persistent chrome.

Acceptance criteria

  • ⌘K (and Ctrl K) from anywhere in the shell opens the palette; the browser's default for that chord does not fire (it's preventDefaulted). Pressing it again while open closes it.
  • On open, the filter input is focused and empty, and the list shows all destinations grouped Go to → Folders → Projects (Go-to = Tasks, Inbox, Priority, Completed, Code, Backlog).
  • Every seeded folder appears as a Folders row (→ /folders/{id}); every seeded project appears as a Projects row (→ /code/{id}) with its KEY pill.
  • Typing filters case-insensitively across all groups; group headers remain for any group with ≥1 match; label-prefix matches sort above label-substring matches; a project matches on its key as well as its name.
  • ↑/↓ move the active row across the flattened list; hovering a row makes it active; ↵ navigates to the active row's destination and closes the palette; Esc (and overlay click) closes it without navigating.
  • Selecting a destination performs a client-side pushState (no full reload/RSC round-trip) and the corresponding view renders; selecting a Code destination shows the Code module, a Tasks destination the Tasks module.
  • The palette is a labelled combobox/listbox (accessible name via a visually-hidden DialogTitle, aria-activedescendant tracks the active row); it is keyboard-operable end to end and honors prefers-reduced-motion (inherited from FormDialog/overlay classes).
  • A quiet ⌘K hint is visible somewhere in the persistent chrome so the shortcut is discoverable.
  • ⌘P content search is unchanged and still works independently.
  • Behavior is pinned by tests at the right level — unit (command-destinations.test.ts: empty-query-shows-all, filtering, prefix-before-substring ranking, project-key match, grouping, href correctness), RTL (command-palette.test.tsx: open/filter/keyboard-select/close), and E2E (command-palette.spec.ts: ⌘K → type a folder/project name → ↵ lands on that view; Esc closes). npm run check is green.
  • A demo doc at docs/demos/alf-3-cmd-k-nav/… captures the open → filter → navigate flow with a screenshot of the running palette, per the showboat skill, and npm run demo -- verify passes.

Out of scope / open questions

  • Content search stays on ⌘P. ⌘K lists destinations, never individual task/story rows — no overlap, no merge, and ⌘P is untouched.
  • Command actions (e.g. "New folder", "Complete task", "Sign out") are a natural future extension of a palette but are not in this ticket — navigation only. The Go to grouping leaves room to add an Actions group later.
  • Recency / frecency ordering and fuzzy (non-substring) scoring are out; ranking is the simple prefix/substring ladder reused from ⌘P.
  • Per-group "+N more" caps are unnecessary at MVP scale and deferred; the RESULTS_PER_GROUP pattern is the template if ever needed.
  • Mobile trigger: ⌘K is keyboard-first; mobile users already have the hamburger drawer for the same destinations, so no touch trigger button is added. The palette itself remains responsive if opened.
  • Open question (deferrable): should the discoverability ⌘K hint also be a clickable trigger (mouse users)? Default assumption: hint only, not clickable — revisit if desired. Making it clickable would mean lifting open into a tiny store like SearchProvider.
