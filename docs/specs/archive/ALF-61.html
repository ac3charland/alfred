Refinement spec · phase: refinement

ALF-61 — Add "Review PR" buttons to cards in Refinement / Ready for Review stages when they exist

Surface a one-click Review PR link on a story card while its work is awaiting human review, so a reviewer can jump straight to the open pull request from the board — without opening the detail modal first.

Context & problem

The code board (frontend/components/code/board.tsx → swimlane.tsx → story-card.tsx) renders each story as a compact card. Today a card only grows an action in the two launchable states: a Refine (+ subordinate Skip to Development) chip in needs_refinement, and an Implement chip in ready_for_dev (see launchPhasesFor in frontend/lib/code/launch.ts). In every other state the card is just ref + title.

That leaves a gap in the two states where the human's job is literally to review an open PR:

  • In Refinement (in_refinement) — a refinement session has opened its spec PR; the human needs to review that refinement_pr_url.
  • Ready for Review (ready_for_review) — an implementation session has opened its PR; the human needs to review that implementation_pr_url.

Both URLs already exist on the story row. CodeStory is the v_code_stories view row, which carries refinement_pr_url and implementation_pr_url (populated by the webhook Worker and already mapped in the store, frontend/lib/stores/code-store.tsx). The detail modal already links them via PrLink (frontend/components/code/story-detail-modal.tsx lines 223–228). But you have to open the modal to get there. ALF-61 brings that link up onto the card itself, so the board is actionable at a glance.

Proposed change

Add a Review PR chip to the card's action footer (the same flex flex-wrap gap-1.5 row that today holds the launch chips) in the two review states, only when the state's PR URL is present. The chip is an external link (opens the GitHub PR in a new tab), styled as a card chip and carrying the ExternalLink icon already used by the modal's PrLink.

Where the logic lives

Mirror the existing launchPhasesFor pattern with a small, pure, unit-tested selector rather than inlining the state→url branching in the component. Proposed new module frontend/lib/code/review-pr.ts: 

import type { CodeStory } from '@/lib/types';

/**
 * The open PR a reviewer should look at for a story in a *review* state, or null.
 * `in_refinement` → the refinement (spec) PR; `ready_for_review` → the implementation PR.
 * Any other state (or a review state whose PR isn't recorded yet) returns null, so the card
 * renders no Review PR chip.
 */
export function reviewPrUrlFor(story: CodeStory): string | null {
  if (story.factory_state === 'in_refinement') return story.refinement_pr_url;
  if (story.factory_state === 'ready_for_review') return story.implementation_pr_url;
  return null;
}

The chip atom

Add a presentational atom frontend/components/atoms/review-pr-chip.tsx, matching the LaunchButton chip's placement and geometry but rendered as an anchor (it navigates, it doesn't launch a session). It reuses the modal's icon-plus-label idiom for consistency:

  • Renders <a href={url} target="_blank" rel="noreferrer"> with an ExternalLink icon (size={12}) and the text Review PR.
  • Chip chrome consistent with the launch chip: inline-flex items-center gap-1.5 rounded-md border px-2 py-1 text-xs font-medium, teal/blue accent, and the same focus-visible:ring-2 focus-visible:ring-accent-blue focus ring the card controls use.
  • Accessible name is simply “Review PR”. If we want it screen-reader unambiguous across many cards, include the ref via aria-label={`Review PR ${ref}`} (decide during implementation; a plain "Review PR" name is acceptable since each card is its own landmark).

Wiring into the card

In frontend/components/code/story-card.tsx, compute const reviewPrUrl = reviewPrUrlFor(story) and render the chip inside the same footer container that holds the launch chips (kept outside ClickableCard so clicking it does not fire the card's onOpen, exactly like the launch chips today). Render the footer when either there are launch phases or a review PR URL exists: 

const reviewPrUrl = reviewPrUrlFor(story);
// ...
{phases.length > 0 || reviewPrUrl !== null ? (
  <div className="flex flex-wrap gap-1.5 px-3 pb-2">
    {phases.map((phase) => (
      <LaunchButton key={phase} story={story} phase={phase} onOpenSession={onOpenSession} variant="chip" />
    ))}
    {reviewPrUrl !== null ? <ReviewPrChip url={reviewPrUrl} /> : null}
  </div>
) : null}
The two review states never return launch phases, so in practice the footer holds either launch chips or the Review PR chip, never both — but the condition is written defensively so a future state that has both would still render correctly.

Real-time behavior (no refresh needed)

The chip must appear the moment a refinement/implementation session opens its PR — no page reload. This already works with zero additional wiring, because the chip is a pure function of two fields the store already keeps live:

  • CodeProvider subscribes to postgres_changes on the base code_items table (frontend/lib/stores/code-store.tsx, the realtime effect around lines 390–450). The webhook Worker writes the PR url (and the review-state transition) out of band; each UPDATE is fed through codeItemToStoryPatch.
  • codeItemToStoryPatch (lines 312–328) already projects both refinement_pr_url and implementation_pr_url (plus factory_state) onto the story, and patchStory dispatches the patch keyed by item_id. React re-renders the affected StoryCard, reviewPrUrlFor now returns the URL, and the chip mounts.

So the sequence "story enters in_refinement → session opens spec PR → Worker writes refinement_pr_url → realtime UPDATE → chip appears" happens with no user action. The implementation must simply not gate the chip on anything the realtime patch doesn't refresh (it doesn't — it reads only story.factory_state and the two url fields, all covered by the patch), and must add a store-level regression test proving it:

  • In frontend/lib/stores/code-store.test.tsx (which already has a realtime UPDATE harness and a codeItemToStoryPatch block asserting the pr-url fields at lines ~1676–1693), add a test that renders a board card in in_refinement with refinement_pr_url: null (no chip), drives a realtime UPDATE carrying the same state + a non-null refinement_pr_url, and asserts the Review PR link now appears — without a re-render triggered by anything other than the patch.
Nothing in the store, the subscription, or codeItemToStoryPatch needs to change for real-time to work — this section is a guarantee to preserve and test, not new work. If a future refactor ever drops the pr-url fields from codeItemToStoryPatch, the chip would silently stop updating live; the regression test above guards against that.

Acceptance criteria

  • A card in in_refinement with a non-null refinement_pr_url shows a Review PR chip whose href is that URL, opening in a new tab.
  • A card in ready_for_review with a non-null implementation_pr_url shows a Review PR chip whose href is that URL, opening in a new tab.
  • A card in in_refinement or ready_for_review whose relevant PR URL is null shows no Review PR chip (ref + title only).
  • No Review PR chip renders in any other state (needs_refinement, ready_for_dev, in_development, done, blocked, abandoned) — even if a PR URL happens to be populated.
  • Clicking the Review PR chip does not fire the card's onOpen (the detail modal does not open), matching the launch chips' behavior.
  • The chip is keyboard-focusable and carries the accent focus ring; existing launch-chip and card-open behavior is unchanged.
  • reviewPrUrlFor is covered by unit tests for all three branches (in_refinement, ready_for_review, other) including the null-url fallthrough.
  • When a story already on the board receives its PR url out of band (a realtime code_items UPDATE), the Review PR chip appears without a page refresh — proven by a store-level test that drives the realtime handler.
  • Storybook snapshot baselines include the two new "with PR" card states (approved via the storybook capture-then-approve flow).
  • npm run check is green.

Out of scope / open questions

  • No in_development chip. An implementation PR may already be open while a story is still in_development, but ALF-61 explicitly scopes the chip to the two review stages. Deferred; revisit if reviewers want the link earlier.
  • No changes to the detail modal. The modal keeps its existing PrLink rows ("Refinement PR" / "Implementation PR"); this ticket only adds the board-card affordance. (A later cleanup could share one chip/link primitive across card and modal, but that's not required here.)
  • Label wording. Spec assumes "Review PR" for both states (the reviewer's verb is the same regardless of which PR). If a per-state label ("Review spec PR" / "Review PR") is preferred, flag it in refinement review.
  • No new data. Relies entirely on refinement_pr_url / implementation_pr_url already being populated by the webhook Worker; this ticket does not touch that pipeline.
