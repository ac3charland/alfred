ALF-68 — Make ticket creation toasts clickable to navigate to story

When a task is admitted to the Software Factory, the confirmation toast (“Created ALF-42”) becomes a link: one click deep-links straight to the new story's board modal, instead of leaving the user to hunt for the freshly-created card.

Context & problem

Sending a task through the gate (frontend/components/code/gate-dialog.tsx) fires a toast via the shared toast store. The single-item path (frontend/components/tasks/task-row.tsx) toasts Created <ref>; the bulk path (frontend/components/tasks/inbox-bulk-bar.tsx) toasts Sent N items to Code. Today those toasts are inert text — the user knows the story exists but has to switch to the Code module and find the card themselves.

The board already has a deep-link seam: frontend/components/code/board.tsx resolves a ?story=<ref> query param into an open StoryDetailModal, and backlog-row.tsx already links rows to /code/<projectId>?story=<ref> through the client-side ViewLink. ALF-68 reuses exactly that mechanism from the creation toast.

Proposed change

Give the toast store an optional navigation target and render a clickable toast as a link.

  • Toast store (frontend/lib/stores/toast-store.tsx) — add an optional href to the Toast type and a third href argument to showToast(message, variant?, href?). It is spread conditionally so the property stays absent under exactOptionalPropertyTypes when omitted.
  • Toast viewport (frontend/components/shell/toast-viewport.tsx) — when a toast has an href, render its message as a ViewLink (a sibling of the close button, never a wrapper, so there are no nested interactive elements). Clicking it navigates client-side and dismisses the toast in one go (ViewLink runs onClick before history.pushState). A toast with no href keeps its plain <span> — unchanged.
  • Creation site (frontend/components/tasks/task-row.tsx) — the reconciled CodeStory already carries project_id and ref, so build /code/<project_id>?story=<ref> and pass it to showToast.

Toast → destination

ToastFired fromClick destination
Created <ref> single convert (task-row.tsx) /code/<project_id>?story=<ref> (opens the story modal)
Sent N items to Code bulk send (inbox-bulk-bar.tsx) — (unchanged; no single-story target)
Scope decision. The ticket says navigate to story (singular). Only the single-item Created <ref> toast maps cleanly to one story, so only it becomes clickable. The bulk Sent N items toast has no single-story target and is left as plain text; the store's new href field makes extending it (e.g. to the project board) a one-line change later.

Acceptance criteria

  • After a single task is sent to Code, the Created <ref> toast is a link whose href is /code/<project_id>?story=<ref>.
  • Clicking that toast navigates client-side to the story's board and dismisses the toast.
  • A toast with no href (every error toast, the code-move toast, the bulk send toast) still renders as plain, non-link text — no regression.
  • The link is a sibling of the dismiss button, so there are no nested interactive elements; the close button still dismisses.
  • showToast stores an href when passed and leaves it absent otherwise (unit-tested).
  • npm run check is green.

Testing & demo

  • Component — toast-viewport.test.tsx: a toast with an href renders a link with that href; without one it stays plain text; clicking the link calls history.pushState with the href and dismisses the toast.
