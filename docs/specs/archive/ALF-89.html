ALF-89 — Refine drag gestures on mobile (hold-first)

Stop touch scroll gestures from being mis-read as task drags. On a touch device, a drag should require a brief press-and-hold before it engages; a plain swipe must scroll the list as normal.

Module: Tasks Area: dnd-kit sensors Type: bug / UX refinement No schema / API change

1 · Context & problem

In the Tasks module the entire task row is a drag surface — there is no dedicated drag handle. Dragging a row re-parents the task, files it into a folder, or promotes a subtask to the top level (see components/tasks/task-dnd-provider.tsx). Drag activation is governed by a single custom pointer sensor: 

// frontend/components/tasks/task-dnd-provider.tsx
const sensors = useSensors(
  useSensor(RowPointerSensor, { activationConstraint: { distance: 8 } }),
  useSensor(RowKeyboardSensor),
);

RowPointerSensor extends dnd-kit's PointerSensor, which handles mouse and touch through the unified pointer-events model. The { distance: 8 } constraint means: as soon as the pointer moves 8px, a drag begins.

The bug. On a phone/tablet, the natural way to scroll the task list is to press a row and swipe. That swipe moves the pointer well past 8px almost instantly, so PointerSensor activates a drag and calls preventDefault on the move — the page never scrolls, and the user is left dragging a task they only meant to scroll past. The list is effectively un-scrollable by touch.

The ticket title captures the intended fix: “hold first, or something.” The correct, idiomatic dnd-kit remedy is to give touch a press-and-hold activation while leaving mouse untouched.

2 · Root cause, precisely

A single PointerSensor cannot behave differently for a mouse than for a finger — its one activationConstraint applies to every input. A distance constraint is exactly right for a mouse (a small, deliberate drag) but exactly wrong for touch (indistinguishable from the start of a scroll). We need per-input-type activation: distance for mouse, delay-with-tolerance for touch.

Why native scroll still works today (and must keep working). The rows do not set touch-action: none anywhere (verified across the frontend). That means the browser still owns vertical scrolling by default — the only thing stealing the gesture is the sensor's preventDefault once a drag activates. Fix the activation and native scroll returns for free. Do not add touch-action: none to task rows — it would disable scrolling on the row entirely, which is the opposite of the goal.

3 · Proposed change

Replace the single unified pointer sensor with the canonical dnd-kit split: a mouse sensor and a touch sensor, each carrying the existing “don't start a drag from a button / input” guard (isInteractiveTarget). dnd-kit routes each input to the sensor that listens for it (MouseSensor → mousedown, TouchSensor → touchstart), so they coexist under one useSensors and each gets its own activation rule. This also handles hybrid devices (touchscreen laptop) correctly — each input type is routed independently, with no viewport/media-query guessing.

3.1 · New / changed sensor classes

lib/dnd/pointer-sensor.ts currently exports RowPointerSensor (extends PointerSensor) and the shared isInteractiveTarget guard. Split the sensor in two, keeping isInteractiveTarget as the single shared guard (do not duplicate it):

SensorExtendsActivation constraintGuard
RowMouseSensor MouseSensor { distance: 8 } — unchanged mouse feel: a small deliberate drag lifts the row. isInteractiveTarget in the onMouseDown activator (reject drags from buttons/inputs).
RowTouchSensor TouchSensor { delay: 250, tolerance: 5 } — hold the row still for ~250ms to lift it; move >5px within that window and no drag starts (the browser scrolls instead). isInteractiveTarget in the onTouchStart activator.
RowKeyboardSensor KeyboardSensor unchanged (Space/Enter to lift, arrows to move). existing isInteractiveTarget guard — unchanged.

Each new activator mirrors the existing RowPointerSensor one — same primary-button / primary-touch check, same isInteractiveTarget early-out, same options.onActivation?.({ event }) call-through so this stays bound. Only the base class and the eventName differ. 

// lib/dnd/pointer-sensor.ts — shape of the touch activator (mirrors the mouse one)
export class RowTouchSensor extends TouchSensor {
  static override activators = [
    {
      eventName: 'onTouchStart' as const,
      handler: ({ nativeEvent: event }: TouchEvent, options): boolean => {
        if (isInteractiveTarget(event.target)) return false;
        options.onActivation?.({ event });
        return true;
      },
    },
  ];
}

File naming is the implementer's call — either keep both mouse & touch sensors in lib/dnd/pointer-sensor.ts (rename considered but not required) or add lib/dnd/touch-sensor.ts alongside it. The shared isInteractiveTarget stays exported from one place.

3.2 · Provider wiring

// frontend/components/tasks/task-dnd-provider.tsx
const sensors = useSensors(
  - useSensor(RowPointerSensor, { activationConstraint: { distance: 8 } }),
  + useSensor(RowMouseSensor, { activationConstraint: { distance: 8 } }),
  + useSensor(RowTouchSensor, { activationConstraint: { delay: 250, tolerance: 5 } }),
  useSensor(RowKeyboardSensor),
);

Everything downstream — collision detection (pointerWithinVisible), the drag-end resolution (folder / promote / re-parent), the DragOverlay — is unchanged. Sensors only decide when a drag starts; they don't touch how it resolves.

4 · Considered alternatives

OptionVerdict
Keep one PointerSensor; pick the constraint via useMediaQuery('(pointer: coarse)') — delay+tolerance when coarse, distance otherwise. Rejected. Smaller diff, but the media query keys off the primary pointer, so a hybrid touchscreen-laptop with a mouse reports fine and gets the distance constraint for touch too — the scroll bug survives on that device. The MouseSensor+TouchSensor split routes each input independently and needs no viewport guessing. Noted as the fallback if the split proves noisy.
Give the whole row a delay+tolerance constraint for mouse and touch alike (one PointerSensor, { delay, tolerance }). Rejected. Regresses desktop feel — mouse users would have to pause before every drag. We only want the hold on touch.
Add a dedicated drag handle (grip icon) and drag only from it. Out of scope. A larger visual/interaction redesign (every row gains a handle; the “whole row is the surface” model is discarded). If the hold-first gesture proves unsatisfying in practice, raise it as a follow-up ticket — not this fix.

5 · Acceptance criteria

  • Touch scroll is restored. On a touch device, pressing a task row and swiping vertically scrolls the list — no drag starts, no DragOverlay appears, no task moves.
  • Touch drag still works, hold-first. Pressing a row and holding it still for ~250ms, then moving, lifts the row and lets it be dropped onto a folder / another task / the promote zone exactly as before.
  • Mouse behaviour is unchanged. With a mouse, a row still lifts after an ~8px drag — no hold required, no perceptible change from today.
  • Keyboard drag is unchanged. Space/Enter to lift, arrows to move, Space to drop, Esc to cancel — the RowKeyboardSensor path is untouched.
  • Interactive controls still click. On both mouse and touch, tapping a row's checkbox, chevron, add-subtask, or kebab performs its action and never starts a drag (the isInteractiveTarget guard is carried into both new sensors).
  • Inline title edit is unaffected. Opening and typing in a row's inline title input (incl. spaces) never triggers a phantom drag.
  • No touch-action: none is added to task rows.
  • Guardrails green. check:fast and check:slow pass; no --no-verify, no config weakening, no disable directives.

6 · Testing & demo evidence

Unit (fast tier)

Mirror the existing lib/dnd/pointer-sensor.test.ts pattern: reach into RowTouchSensor.activators[0].handler with a hand-built touch-start event and assert (a) a plain press fires onActivation / returns true, and (b) a press whose target isInteractiveTarget (a <button>) returns false and does not activate. Do the same for RowMouseSensor if the mouse activator diverges from the old pointer one. These prove the guard; the delay/tolerance timing is dnd-kit's own runtime concern configured in the provider, not our handler.

E2E (slow tier, real Chromium)

Add a Playwright spec run with touch emulation (test.use({ hasTouch: true, isMobile: true })) that proves the two touch behaviours the unit test can't:

  • Swipe scrolls, doesn't drag: dispatch a touch press + immediate move (a swipe) on a row and assert the list scrolled and no drag overlay appeared / the task did not change parent or folder. Use page.touchscreen / CDP Input.dispatchTouchEvent for the touch sequence (mouse events won't drive TouchSensor).
  • Hold-then-move drags: touch down, wait >250ms with the finger still, then glide to a folder and release; assert the task is filed (the existing drag-to-folder assertions, driven by touch).

See the dnd-kit skill's “Testing under alfred's gates” notes — jsdom can't measure layout, so drag outcomes belong in E2E, not RTL. Reuse the existing e2e/support/drag.ts helpers where possible; add a touch variant of the pickUp helper if needed (retry the press until the row enters its opacity-40 dragging state to dodge the hydration race).

7 · Out of scope / open questions

  • Dedicated drag handle. Not part of this fix. Only revisit if hold-first proves unsatisfying in real use (would be a new ticket).
  • Tuning the numbers. delay: 250ms and tolerance: 5px are dnd-kit's recommended touch defaults and the proposed starting values. They are tunable during implementation/QA on a real device — adjust if a 250ms hold feels too slow or a 5px tolerance too twitchy. Any change stays within the same mechanism.
  • Haptics / long-press affordance. A subtle visual or haptic cue that the hold “took” (row lifts, subtle scale) could improve discoverability. Deferred — the existing DragOverlay lift is the current signal; propose a separate polish ticket if wanted.
